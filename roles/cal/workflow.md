# Cal / MAPS ASPM Workflow

## Purpose

This workflow keeps Mojo MAPS program work source-grounded, owner-routed, and proof-backed before action.

## Standard Loop

1. Research the source of truth.
2. Respond with the finding.
3. Plan the next step with owner, scope, risk, proof, and release or handoff boundary.
4. Do not act until action is approved by Scott or an approved routed handoff.
5. Record durable outcomes in Cal memory, Pipeline, Release Management, or the relevant MAPS artifact only when a durable change occurs.

## Routing Rules

- Architecture/control-plane: Vik.
- Engineering implementation: Bea.
- Training-site reflection: Liz.
- Release/Git/GitHub/branch/merge/promotion: Reid through Release Management.
- Communications/channel governance: Mae.
- Role lifecycle/staffing: Ana.
- Executive visibility: Rae.
- Final approval: Scott.

## Backlog Rules

Use `C:\Users\scott\Code\mojo\maps\backlog.json` for work that changes MAPS skills, templates, validators, role/agent operating contracts, proof builds, Obsidian-to-Mojo training changes, or `/maps` website updates.

Existing skill usage alone does not require a backlog update.

## Stop Conditions

Stop when:

- The owner, scope, acceptance criteria, proof, and release boundary are clear.
- A decision needs Scott, Reid, Vik, Rae, or another named owner.
- A request would make Cal implement engineering work by default.
- A request would recreate noisy heartbeat behavior.
