M8 / optional A2

Experience Design

Experience Design turns product intent, orchestration, and agent behavior into the user's actual path through the product. Use it as M8 for a multi-agent system, or as optional A2 when a standalone agent is the product.

M8 output

At the end of Experience Design, the required artifact is `m/{project-handle}/experience-design.md` for a multi-agent product or `agents/{agent-handle}/experience-design.md` for optional single-agent A2.

Inputs

M0 Product Intent, M1 System Shape, M2 Roster, M3 Contracts, M4 Coordination, M5 Agent Buildout, M6 Shared Capabilities, M7 Orchestration Build, and any APS artifacts already drafted.

Skill

/design-experience runs the M8 workflow and produces the artifact.

Template

`templates/experience-design.md` captures journeys, surfaces, states, accessibility, and APS handoffs.

Primary handoff

M9 System Evaluate receives product journeys, interaction states, agent visibility rules, accessibility requirements, and user-recognizable success criteria.

Experience design questions

M8 should turn orchestration behavior into explicit user-facing decisions before the system is evaluated and released.

Question Experience design should decide Feeds
User journey How the user gets from intent to successful completion, including pauses, decisions, waiting, and recovery. M9 journey evals, M10 smoke paths
Surfaces Which screens, chat panes, dashboards, notifications, settings, approvals, and review surfaces exist. M9 system scenarios, M10 production surfaces
Information architecture Navigation, object hierarchy, labels, content grouping, history, and wayfinding. M3 contracts, M9 system evals
Agent visibility How planning, tool use, memory use, uncertainty, citations, approvals, refusals, handoffs, and completion appear. M6 shared capabilities, M9 system evals, M10 observation signals
User control Where users can approve, pause, cancel, retry, undo, escalate, or take over. M3 contracts, M4 coordination, M7 orchestration
Visual and tone standard Density, hierarchy, component expectations, language, care text, warning style, and product personality. M9 system evals, M11 improvement backlog
Accessibility Keyboard paths, focus behavior, semantics, contrast, motion, form errors, and readable state copy. M9 accessibility checks, M11 improvement routing

Interaction states

Agent products often fail in the in-between moments. M8 names those moments after orchestration is known and before evaluation treats the product as ready.

Waiting

Loading, streaming, long-running work, queue position, timeouts, partial progress, and background completion.

Control

Approval, edit-before-send, pause, cancel, retry, undo, escalation, and handoff to a human owner.

Trust

Evidence, citations, uncertainty, tool call visibility, memory disclosure, permission boundaries, and refusal explanation.

Failure

Empty, blocked, partial success, recoverable error, unsafe request, missing permission, stale data, and post-incident review.

Self-guided M8 path

01

Read the M inputs

Start with product intent, system shape, roster, contracts, coordination, shared capabilities, orchestration, and any existing APS artifacts.

Check it

M8 should not create a new product, roster, or orchestrator; it makes the orchestrated system usable.

02

Map journeys and surfaces

Draw the core user paths and list every surface, touchpoint, decision, waiting state, and recovery path.

Check it

The first useful slice should be visible in the journeys.

03

Define agent visibility

Name how the product exposes planning, tool use, memory, uncertainty, approvals, refusals, handoffs, and completion.

Check it

Users should know when an agent is acting, waiting, blocked, uncertain, or asking for permission.

04

Lock states and accessibility

Capture state behavior, content rules, keyboard/focus expectations, contrast, motion, and assistive-technology notes.

Check it

Every risky or repeated state should have a clear user action and clear copy.

05

Hand off to system proof

Record what M9 System Evaluate, M10 Deploy/Observe, and M11 Improve must inherit from the experience design.

Check it

System evaluation should prove the user-recognizable experience, not only the internal agent workflow.
