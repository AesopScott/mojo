What Improve means

Improve closes the MAPS loop by converting evidence into prioritized work that can re-enter Design, Build, Equip, Evaluate, Deploy, or Observe.

Evidence, not vibes

Every improvement should point back to an observation, eval failure, incident, support note, user feedback item, or operating signal.

Route before build

Some improvements need design changes, some need tools or permissions, some need eval coverage, and only some should go straight to Build.

Small next slice

Phase 8 turns broad frustration into small backlog items with priority, dependencies, proof required, and a target phase.

Phase 8 output

At the end of Improve, the required artifacts are `agents/{agent-handle}/improvement-review.md` and `agents/{agent-handle}/improvement-backlog.md`.

Input

`observation-log.md`, eval report, deployment record, incident/support notes, user feedback, cost/latency signals, and existing backlog items.

Skill

/improve-agent reviews evidence, classifies improvements, routes work, prioritizes items, and creates the next backlog.

Templates

`templates/improvement-review.md` captures the decision trail. `templates/improvement-backlog.md` captures the ordered work.

Iteration gate

The improvement review should recommend next iteration, defer, monitor, rollback/pause, or no action.

Improve must cover

Phase 8 should not let evidence vanish into vague follow-up. These items must be reviewed before the next MAPS loop starts.

Evidence

Review observation evidence

Start with Phase 7 traces, logs, metrics, incidents, feedback, and observation findings.

Evidence

Review eval failures and regressions

Bring failed eval cases, drift, flaky behavior, and regression history into the improvement review.

Evidence

Review incidents and support notes

Use incident records, support tickets, operator notes, and escalations as improvement sources.

Evidence

Review user feedback

Include ratings, complaints, feature requests, acceptance issues, and real user friction.

Evidence

Review cost, latency, and reliability signals

Use budget pressure, slow runs, failed jobs, rate limits, retries, and availability issues as backlog inputs.

Classify

Classify improvement type

Name whether the item is behavior, prompt/design, tool, permission, memory, runtime config, eval, deployment, observability, cost, reliability, safety, or UX work.

Route

Route each item to the right MAPS phase

Send the work back to Define, Design, Build, Equip, Evaluate, Deploy, Observe, or another Improve pass.

Priority

Prioritize by impact, safety, frequency, cost, dependency, and effort

Make tradeoffs explicit instead of letting the loudest issue automatically become the next build.

Backlog

Split large improvements into smaller backlog items

Break broad changes into independent slices or spikes that can be proved separately.

Proof

Define proof required for each item

Require the eval, trace, smoke test, user check, metric, or manual review that will prove the improvement worked.

Decision

Decide what enters the next iteration

Promote only the highest-priority unblocked items into the next MAPS loop.

Decision

Decide what stays deferred

Keep lower-priority, blocked, speculative, or poorly evidenced work visible without starting it yet.

Classify improvement work

A good improvement backlog says what kind of work each item really is.

Improvement type Common evidence Likely phase
Behavior or prompt/design Wrong answers, misunderstood intent, missing constraints, repeated user correction. Design, then Build or Evaluate.
Tools, permissions, memory, or runtime config Tool failures, missing access, unsafe scopes, stale retrieval, bad fallback behavior. Equip, then Evaluate.
Eval coverage or regression Production failure not covered by tests, unstable score, missing adversarial case. Evaluate.
Deployment or observability Rollback gaps, missing logs, unclear traces, weak alerts, hard-to-debug incidents. Deploy or Observe.
Cost, latency, reliability, safety, or UX Budget spikes, slow runs, failed jobs, boundary violations, user friction. Design, Build, Equip, Evaluate, or Observe depending on root cause.

Route the next loop

Phase 8 decides where improvement work enters the next MAPS pass.

Next iteration

The item has evidence, priority, dependencies, target phase, and proof required. Send it into the next MAPS loop.

Defer

The item is real but lower priority, blocked, missing evidence, or dependent on another improvement.

No action or pause

The item is unsupported by evidence, already resolved, intentionally accepted as a tradeoff, or severe enough to pause/rollback first.

Prioritize the backlog

Improve should produce an ordered backlog, not a dumping ground.

Impact

How many users or workflows does this affect, and how much value is unlocked by fixing it?

Safety

Does the issue create security, privacy, compliance, unsafe action, data exposure, or approval-gate risk?

Frequency

Is this a one-off, a repeated pattern, a predictable edge case, or a growing trend?

Cost and latency

Does the item reduce operating cost, token use, queue time, slow runs, retries, or support workload?

Dependency

Does other important work depend on this item, or is this blocked by a prerequisite?

Effort

Can it be split into a small proof slice, or does it need a design spike before implementation?

Self-guided improve path

01

Gather evidence

Read observation logs, eval failures, incidents, support notes, feedback, and cost/latency/reliability signals.

Check it

Every candidate improvement should have a concrete source.

02

Classify and route

Name the improvement type and route each item to the MAPS phase where it belongs.

Check it

Do not send tool, eval, deployment, or observation problems blindly into Build.

03

Prioritize and split

Order work by impact, safety, frequency, cost, dependency, and effort, then split large items.

Check it

The next item should be small enough to prove independently.

04

Define proof

Record the eval, trace, smoke test, metric, user check, or review that will prove the improvement worked.

Check it

The improvement backlog should say how each item gets verified.

05

Choose the next loop

Decide what enters the next MAPS iteration and what stays deferred.

Check it

Phase 8 should hand off a prioritized, dependency-aware backlog, not an unranked pile.
