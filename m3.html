M3 / optional A2

Experience Design

Experience Design turns product intent, system shape, and agent roster into the user's actual path through the product. Use it as M3 for a multi-agent system, or as optional A2 when a standalone agent is the product.

Build navigation

Review of phases developed so far

The APS layer is usable from Phase 0 through Phase 8. M3 is the first concrete M-layer page and closes the gap between system planning and user-facing build work.

Phase Current strength M3 dependency
Phase 0 Alignment Sets the two-layer MAPS scaffold and phase vocabulary. M3 should keep the M layer separate from APS while sharing resources with optional A2.
Phase 1 Define Defines one agent's job, user, scope, authority, success, failure, and escalation. M3 can send back user-facing constraints, care text, and visibility needs that belong in agent briefs.
Phase 2 Design Turns an agent brief into design decisions, proof gates, runtime target, and build backlog. M3 gives Phase 2 the product journeys, surfaces, IA, state model, and agent visibility rules it needs.
Optional A2 Experience Design Now shares this page when there is no multi-agent M layer. It should run only when the agent is itself the product or ships inside a single-agent surface.
Phase 3 Build Builds from a prioritized backlog, first slice, adapter requirement, and proof plan. M3 prevents Build from inventing UI behavior after implementation has already started.
Phase 4 Equip Maps tools, permissions, memory, connectors, secrets, limits, and fallbacks. M3 names how tools, memory, approvals, refusals, and fallbacks should be visible to users.
Phase 5 Evaluate Builds eval suites and reports with executable or specification modes. M3 contributes journey-level scenarios, accessibility checks, state coverage, and trust failures.
Phase 6 Deploy Packages and releases with runtime metadata, smoke tests, evidence, and rollback notes. M3 clarifies the production surface and the smallest post-deploy smoke path a user can recognize.
Phase 7 Observe Tracks traces, logs, metrics, feedback, incidents, and improvement evidence. M3 defines which user signals, feedback surfaces, and review moments should exist in production.
Phase 8 Improve Turns evidence into prioritized improvement backlog items. M3 helps route UX, IA, content, accessibility, and state-model failures back to the right phase.

M3 output

At the end of Experience Design, the required artifact is `m/{project-handle}/experience-design.md` for a multi-agent product or `agents/{agent-handle}/experience-design.md` for optional single-agent A2.

Inputs

M0 Product Intent, M1 System Shape, M2 Roster, and any APS briefs or designs already drafted.

Skill

/design-experience runs the M3 workflow and produces the artifact.

Template

`templates/experience-design.md` captures journeys, surfaces, states, accessibility, and APS handoffs.

Primary handoff

Phase 2 Design receives constraints for user journeys, interaction model, agent visibility, state behavior, and accessibility.

Experience design questions

M3 should turn vague product feeling into explicit user-facing decisions before agents are built.

Question Experience design should decide Feeds
User journey How the user gets from intent to successful completion, including pauses, decisions, waiting, and recovery. Phase 2 scenarios, Phase 5 journey evals
Surfaces Which screens, chat panes, dashboards, notifications, settings, approvals, and review surfaces exist. Phase 3 build slices, Phase 6 smoke path
Information architecture Navigation, object hierarchy, labels, content grouping, history, and wayfinding. Phase 2 design constraints, M4 contracts
Agent visibility How planning, tool use, memory use, uncertainty, citations, approvals, refusals, handoffs, and completion appear. Phase 4 capability map, Phase 7 observation signals
User control Where users can approve, pause, cancel, retry, undo, escalate, or take over. Phase 1 authority, Phase 2 workflow, M5 coordination
Visual and tone standard Density, hierarchy, component expectations, language, care text, warning style, and product personality. Phase 3 UI implementation, Phase 8 polish backlog
Accessibility Keyboard paths, focus behavior, semantics, contrast, motion, form errors, and readable state copy. Phase 3 implementation proof, Phase 5 checks

Interaction states

Agent products often fail in the in-between moments. M3 names those moments before the implementation invents them ad hoc.

Waiting

Loading, streaming, long-running work, queue position, timeouts, partial progress, and background completion.

Control

Approval, edit-before-send, pause, cancel, retry, undo, escalation, and handoff to a human owner.

Trust

Evidence, citations, uncertainty, tool call visibility, memory disclosure, permission boundaries, and refusal explanation.

Failure

Empty, blocked, partial success, recoverable error, unsafe request, missing permission, stale data, and post-incident review.

Self-guided M3 path

01

Read the M inputs

Start with product intent, system shape, roster, and any existing APS artifacts.

Check it

M3 should not create a new product or roster; it makes the existing one usable.

02

Map journeys and surfaces

Draw the core user paths and list every surface, touchpoint, decision, waiting state, and recovery path.

Check it

The first useful slice should be visible in the journeys.

03

Define agent visibility

Name how the product exposes planning, tool use, memory, uncertainty, approvals, refusals, handoffs, and completion.

Check it

Users should know when an agent is acting, waiting, blocked, uncertain, or asking for permission.

04

Lock states and accessibility

Capture state behavior, content rules, keyboard/focus expectations, contrast, motion, and assistive-technology notes.

Check it

Every risky or repeated state should have a clear user action and clear copy.

05

Hand off to APS

Record what Phase 1, Phase 2, Phase 3, Phase 5, and Phase 7 must inherit from the experience design.

Check it

Phase 3 should not need to invent the product experience from implementation details.
