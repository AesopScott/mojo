Must-Know Rules

Agent Loop

If stop_reason == "tool_use":
  execute requested tool
  append tool result to conversation
  continue loop

If stop_reason == "end_turn":
  stop

Do not terminate by parsing assistant text. Do not use arbitrary iteration cap as primary completion logic.

Multi-Agent Systems

  • Coordinator owns decomposition, routing, error handling, and aggregation.
    Decomposition means breaking one large task into smaller sub-tasks before routing them.
  • Subagents have isolated context.
    Each subagent only sees the context you explicitly give it.
  • Pass all needed context explicitly in subagent prompt.
    Include the facts, constraints, and source material the subagent needs up front.
  • Use structured context with source URLs, document names, page numbers, and metadata.
    That means hand off exact references so the subagent can find and verify the source quickly.
  • Spawn parallel subagents with multiple Task tool calls in one coordinator response.
    Yes, Task is a real tool call, and multiple independent calls can run at once.
  • Coordinator must have allowedTools including Task to spawn subagents.
    The coordinator cannot launch subagents unless Task is enabled in its allowed tools.

Hooks

  • Use hooks when compliance must be deterministic.
    Use the hook name when you want exact scripted behavior, not a vague behavioral hint.
  • PostToolUse can normalize tool results before Claude sees them.
    This is the right place to inspect, rewrite, redact, or reject tool output after execution.
  • Tool-call interception can block policy violations, such as refunds over threshold.
    If the tool runs first and the hook filters the result, that is still post-tool-use interception.

Tools and MCP

  • Tool descriptions drive tool selection.
  • Split vague generic tools into purpose-specific tools.
  • Too many tools degrades selection reliability.
  • Give each subagent only role-relevant tools.
  • MCP tool = an executable action the model can invoke.
    Use tools for things that do work, like creating, updating, searching, sending, or changing state.
  • MCP resource = read-only context the model can inspect.
    Use resources for facts, documents, or data the model should read but not modify.
  • MCP prompt = a prebuilt runtime prompt with fixed instructions and named placeholders.
    The fixed language is already there, and the caller fills in the variable parts when the prompt runs.
  • MCP errors should include isError, category, retryable flag, and human-readable explanation.
    Make errors structured so the model can tell what failed, whether to retry, and how to explain the failure.

tool_choice

OptionMeaning
autoClaude may call a tool or answer normally.
Auto is common for flexible assistants that should decide whether tools are needed. Exploratory or conversational flows often use auto.
anyClaude must call some tool.
Any is less common but useful when you want to guarantee tool usage without caring which one. It is the middle ground when you want tool usage but not a specific tool.
forced toolClaude must call named tool.
Named tool is common when the workflow is specific and you already know the exact tool to call. Scripted or deterministic flows often use a forced tool.

Structured Output

If you want repeatable machine-readable output instead of plain text, build a tool so the runtime can enforce the shape.

  • Use tool_use with input_schema for reliable structured output.
    The schema is what enforces the shape of the tool arguments, so Claude has to fill named fields instead of free-form text. If you merely describe a format in the prompt, the model can still drift. If you use a tool, the runtime enforces the schema for the tool call.
  • Use required vs optional fields intentionally.
    Mark only the fields you truly need as required; leave optional fields open when they may not exist on every case.
  • Use nullable fields instead of forcing hallucinated values.
    If the value can be unknown or missing, let it be null rather than making Claude invent something plausible.
  • Use enums for finite choices.
    Use enums when the field should only accept a closed set of values, like status labels or category names.
  • Use "other" plus detail field for unknown categories.
    Give Claude a safe escape hatch for cases that do not fit your enum, then capture the explanation in a detail field.
  • Validate with Pydantic or another schema validator.
    Run a validator after the model responds so bad shapes or missing fields are caught before downstream code uses them.
  • Retry with validation feedback.
    If validation fails, send the errors back so Claude can repair the output instead of guessing silently.
  • Include field-level confidence and evidence for extraction.
    For each extracted field, store how sure the model is and what source text supports the value.

Message Batches API

  • 50% cost savings.
    Use batches when latency matters less than price, because the tradeoff is cheaper throughput instead of immediate response.
  • Up to 24-hour processing window.
    Batch jobs can take a long time, so they are better for deferred work than live user interactions.
  • Use custom_id to correlate responses.
    Give each request a stable ID so you can match each result back to the original input.
  • No multi-turn tool calling support.
    Use batches for one-shot jobs, not back-and-forth agent loops.
  • Good for overnight reports.
    Batch processing works well when you can submit work, walk away, and review results later.
  • Bad for blocking pre-merge checks.
    Do not use batches when someone needs immediate feedback before continuing.
  • Separate API, same key, workspace scoped.
    Message Batches is its own API path, but it uses the same API key as other API calls. Access is scoped to the workspace, which is the container for your organization’s API access, billing, and resources.

Context and Reliability

  • Use progressive summarization for long sessions.
    Summarize as the task grows so the important state stays visible and the chat does not rot from sheer length.
    - don’t ask for one giant final summary
    - ask the model to keep compressing the running state as the session evolves
  • Avoid lost-in-the-middle by ordering and extracting key facts.
    Put the critical items where they are easiest to find and keep the most important facts near the top.
  • Use scratchpad files for long codebase exploration.
    Move working notes into files when the investigation gets too large for chat alone.
  • Do not paste whole repos into context.
    Bring only the relevant files, snippets, or summaries so the model can focus on what matters.
  • Escalation needs explicit criteria and structured handoff.
    Define exactly when a problem gets handed off and what evidence the next person needs.
  • Preserve claim-source mappings.
    Keep every important claim tied back to the source that supports it.
  • Annotate conflicts and coverage gaps.
    Call out disagreements and missing areas so nobody mistakes partial evidence for full confidence.
  • Calibrate confidence with labeled validation sets and stratified sampling.
    Measure performance against balanced test data so confidence reflects reality instead of a lucky subset.

Live Drill Set

Answer with the specific primitive or design choice.

  1. Pipeline hangs waiting for input.
    Use claude -p so the pipeline runs non-interactively and exits with a result instead of waiting on a human.
  2. Refund must never happen before identity verification.
    Use a programmatic prerequisite or hook, not prompt-only guidance, so the check is enforced before the refund action runs.
  3. Synthesis subagent needs web findings.
    Pass the findings explicitly in the prompt with source metadata so the subagent sees the evidence it is supposed to synthesize.
  4. MCP timeout occurs during research.
    Return a structured error with category, retryable flag, attempted actions, and partial results so the coordinator can decide what to do next.
  5. CI output must be machine-readable.
    Use --output-format json and --json-schema so the pipeline can parse the output reliably.
  6. Extraction hallucinates missing field.
    Use a nullable field, require evidence, and retry validation so the model can leave the value empty instead of inventing one.
  7. Large PR review misses cross-file bugs.
    Do per-file passes first, then add a cross-file integration pass so local correctness does not hide system-level bugs.
    Build a cross-boundary audit to verify registries, manifests, schemas, and other required artifacts exist before production.
  8. Technical debt report due tomorrow morning.
    Use the Message Batches API because the job is deferred, cost-sensitive, and does not need an immediate response.
  9. Blocking PR review before merge.
    Use a real-time call, not batch processing, because this needs fast feedback before the merge happens.
  10. Report has conflicting dates from sources.
    Preserve both sources, annotate the conflict, and report uncertainty instead of pretending one date is certain.

Exam Traps

  • Subagents do not automatically inherit parent context.
  • Prompt instructions are not deterministic enforcement.
  • Empty result is not same as failed query.
  • Larger context window does not fix attention dilution.
  • More tools can make tool selection worse.
  • Batch API is wrong for latency-sensitive blocking work.
  • claude -p is the CI/non-interactive flag.
  • MCP resource is not an action.
  • Hooks are for deterministic event/tool interception, not model judgment.
  • Prompt caching internals are out of scope beyond knowing it exists.
