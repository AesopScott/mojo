Must-Know Rules

Agent Loop

If stop_reason == "tool_use":
  execute requested tool
  append tool result to conversation
  continue loop

If stop_reason == "end_turn":
  stop

Do not terminate by parsing assistant text. Do not use arbitrary iteration cap as primary completion logic.

Multi-Agent Systems

  • Coordinator owns decomposition, routing, error handling, and aggregation.
    Decomposition means breaking one large task into smaller sub-tasks before routing them.
  • Subagents have isolated context.
    Each subagent only sees the context you explicitly give it.
  • Pass all needed context explicitly in subagent prompt.
    Include the facts, constraints, and source material the subagent needs up front.
  • Use structured context with source URLs, document names, page numbers, and metadata.
    That means hand off exact references so the subagent can find and verify the source quickly.
  • Spawn parallel subagents with multiple Task tool calls in one coordinator response.
    Yes, Task is a real tool call, and multiple independent calls can run at once.
  • Coordinator must have allowedTools including Task to spawn subagents.
    The coordinator cannot launch subagents unless Task is enabled in its allowed tools.

Hooks

  • Use hooks when compliance must be deterministic.
    Use the hook name when you want exact scripted behavior, not a vague behavioral hint.
  • PostToolUse can normalize tool results before Claude sees them.
    This is the right place to inspect, rewrite, redact, or reject tool output after execution.
  • Tool-call interception can block policy violations, such as refunds over threshold.
    If the tool runs first and the hook filters the result, that is still post-tool-use interception.

Tools and MCP

  • Tool descriptions drive tool selection.
  • Split vague generic tools into purpose-specific tools.
  • Too many tools degrades selection reliability.
  • Give each subagent only role-relevant tools.
  • MCP tool = an executable action the model can invoke.
    Use tools for things that do work, like creating, updating, searching, sending, or changing state.
  • MCP resource = read-only context the model can inspect.
    Use resources for facts, documents, or data the model should read but not modify.
  • MCP prompt = a prebuilt runtime prompt with fixed instructions and named placeholders.
    The fixed language is already there, and the caller fills in the variable parts when the prompt runs.
  • MCP errors should include isError, category, retryable flag, and human-readable explanation.
    Make errors structured so the model can tell what failed, whether to retry, and how to explain the failure.

Claude Code

NeedUse
Shared repo/project instructionsCLAUDE.md
Use this for project-wide rules that apply by default, like repo conventions or deployment expectations. Example: "always push production changes through main."
Path-scoped conventions.claude/rules/ with YAML frontmatter/globs
Use this for rules that only apply inside certain folders or file types. Example: apply one rule to architect/** and a different one to forum/**.
Reusable slash workflow.claude/commands/
Use this when you want a repeatable command the user can invoke with a slash. Example: a saved workflow for generating a report.
Reusable task capability.claude/skills/ with SKILL.md
Use this for a reusable capability with instructions, inputs, and steps. Example: a skill for reviewing docs or editing slides.
Complex/architectural changeplan mode
Use this when the work has multiple steps, tradeoffs, or hidden risk. Example: redesigning a workflow or refactoring a system boundary.
Simple local changedirect execution
Use this for a small, obvious edit that does not need planning. Example: fixing a label, link, or short text block.
CI non-interactive runclaude -p / --print
Use this when a script or CI job should run Claude without an interactive session. Example: generate a summary in a pipeline.
Structured CI output--output-format json, --json-schema
Use this when downstream automation needs machine-readable output. Example: a build step that validates fields before continuing.
Long session cleanup/compact
Use this when the conversation is getting long and you want to preserve the important state. Example: before continuing a large task after many turns.
Persistent useful project knowledge/memory
Use this to save durable project facts or preferences for later sessions. Example: store naming rules, deployment notes, or workflow norms.

Use this to reopen a prior Claude session and keep working in the same thread of thought. Example: resume an investigation instead of starting over.

Use this when you want to branch into two different solution paths. Example: try one approach in one fork and a second approach in another.

Structured Output

  • Use tool_use with JSON Schema for reliable structured output.
  • Use required vs optional fields intentionally.
  • Use nullable fields instead of forcing hallucinated values.
  • Use enums for finite choices.
  • Use "other" plus detail field for unknown categories.
  • Validate with Pydantic or equivalent.
  • Retry with validation feedback.
  • Include field-level confidence and evidence for extraction.

Message Batches API

  • 50% cost savings.
  • Up to 24-hour processing window.
  • Use custom_id to correlate responses.
  • No multi-turn tool calling support.
  • Good for overnight reports.
  • Bad for blocking pre-merge checks.

Context and Reliability

  • Use progressive summarization for long sessions.
  • Avoid lost-in-the-middle by ordering and extracting key facts.
  • Use scratchpad files for long codebase exploration.
  • Do not paste whole repos into context.
  • Escalation needs explicit criteria and structured handoff.
  • Preserve claim-source mappings.
  • Annotate conflicts and coverage gaps.
  • Calibrate confidence with labeled validation sets and stratified sampling.

Live Drill Set

Answer with the specific primitive or design choice.

  1. Pipeline hangs waiting for input.

    2. - claude -p.

  1. Refund must never happen before identity verification.

    2. - Programmatic prerequisite/hook, not prompt-only guidance.

  1. Synthesis subagent needs web findings.

    2. - Pass findings explicitly in prompt with source metadata.

  1. MCP timeout occurs during research.

    2. - Structured error with category, retryable flag, attempted actions, partial results.

  1. CI output must be machine-readable.

    2. - --output-format json and --json-schema.

  1. Extraction hallucinates missing field.

    2. - Nullable field, evidence requirement, validation retry.

  1. Large PR review misses cross-file bugs.

    2. - Per-file passes plus cross-file integration pass.

  1. Technical debt report due tomorrow morning.

    2. - Message Batches API is appropriate.

  1. Blocking PR review before merge.

    2. - Real-time call, not batch.

  1. Report has conflicting dates from sources.

    2. - Preserve both sources, annotate conflict, report uncertainty.

Exam Traps

  • Subagents do not automatically inherit parent context.
  • Prompt instructions are not deterministic enforcement.
  • Empty result is not same as failed query.
  • Larger context window does not fix attention dilution.
  • More tools can make tool selection worse.
  • Batch API is wrong for latency-sensitive blocking work.
  • claude -p is the CI/non-interactive flag.
  • MCP resource is not an action.
  • Hooks are for deterministic event/tool interception, not model judgment.
  • Prompt caching internals are out of scope beyond knowing it exists.

Closeout

When stuck, choose the answer that makes the system more:

  • observable
  • recoverable
  • source-faithful
  • policy-compliant
