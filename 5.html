What Evaluate means

Evaluate proves the equipped agent before release by turning expected behavior, tool boundaries, failures, and regressions into repeatable evidence.

Behavior proof

Show that the agent can complete its core loop with the tools, memory, permissions, and runtime from Phase 4.

Safety proof

Show that approval gates, forbidden actions, scopes, budgets, and fallback rules are respected under pressure.

Release evidence

Capture datasets, traces, commands, failures, and decisions so Deploy and Observe inherit real proof.

Phase 5 output

At the end of Evaluate, the required artifacts are `agents/{agent-handle}/eval-suite.md` and `agents/{agent-handle}/eval-report.md`.

Input

`agents/{agent-handle}/capability-map.md` from Phase 4, plus the working agent and target runtime.

Observe handoff

Evaluate coverage

The eval suite should recommend coverage first, then ask the user to override gaps, risk tolerance, runtime limits, or cost.

Eval area What it proves Evidence to capture
Core loop success The agent completes the job defined in Phase 1 and implemented in Phase 3. Inputs, outputs, pass/fail result, trace, command, and reviewer notes.
Tool trajectory The agent selects the right tools, in the right order, with acceptable arguments. Message sequence, tool calls, tool inputs, tool outputs, and trajectory score.
Capability boundaries The agent respects approvals, forbidden actions, scopes, budgets, and runtime limits. Boundary scenarios, blocked actions, approval requests, and escalation behavior.
Failure behavior The agent retries, degrades, asks, escalates, or stops correctly when dependencies fail. Injected failures, fallback decisions, final user-facing behavior, and incident notes.
Regression The same checks can run again when prompts, models, tools, skills, or adapters change. Dataset version, baseline result, current result, and release-gate comparison.

The /evaluate-agent++ wrapper

Phase 5 uses one big wrapper: the base Evaluate process plus LangSmith, Inspect AI, and Phoenix.

Wrapper

LangSmith

Use for datasets, traces, trajectory review, regression history, experiments, and Phase 7 monitoring handoff. LangGraph is not required.

Wrapper

Inspect AI

Use for structured task evals, solvers, scorers, model-graded checks, tool-use checks, and safety or boundary tests.

Wrapper

Phoenix

Use for open-source tracing, LLM eval analysis, RAG or retrieval evals, and a practical bridge into Observe.

Base Evaluate

Owns MAPS artifacts, eval recommendations, user overrides, release gates, and evidence structure.

No LangGraph dependency

LangGraph belongs later when multi-agent orchestration needs graph state, checkpoints, routing, or durable coordination.

Phase 5/7 bridge

Evaluate creates traces and datasets that Observe can keep using for production monitoring and improvement loops.

Execution modes

`/evaluate-agent++` should evaluate the real built agent whenever possible. When it cannot run the agent yet, it must make the missing runtime requirements explicit.

Executable mode

The agent can run locally, in a sandbox, or in the target runtime. `/evaluate-agent++` runs eval cases against the actual built and equipped agent, then records traces, tool calls, outputs, failures, scores, and release evidence.

Specification mode

The agent cannot run yet because secrets, runtime access, connectors, credentials, or deployment pieces are missing. `/evaluate-agent++` still creates the eval suite and report shell, but records exact commands, missing requirements, fixtures, and manual checks needed to run later.

Do not fake it

Do not score a description of the agent as if the agent ran. The report must distinguish executed evidence from planned checks and blocked runtime requirements.

Release gates

Phase 5 ends with a decision, not a pile of test notes.

Pass

The agent meets the required evals, respects boundaries, and has enough evidence to move into Deploy.

Conditional

The agent can move forward only with documented limits, manual gates, deferred risks, or narrow rollout conditions.

Blocked

The agent fails core behavior, safety boundaries, tool behavior, or runtime reliability enough to stop deployment.

Self-guided evaluate path

01

Read and research

Start from the capability map, working agent, target runtime, and comparable eval patterns.

Check it

The recommendation should cite which wrapper pieces it is using and why.

02

Recommend evals

Recommend task, trajectory, boundary, failure, memory, runtime, and regression evals before asking broad questions.

Check it

Each eval should name expected behavior, harness, evidence, and release-gate impact.

03

Choose execution mode

Use Executable mode when the real agent can run. Use Specification mode only when missing secrets, connectors, runtime access, or deployment pieces block execution.

Check it

The report should distinguish executed evidence from planned checks.

04

Gate and hand off

Record pass, conditional pass, blocked, or needs more evaluation, then prepare Observe handoff notes.

Check it

Phase 6 and Phase 7 should inherit the decision, traces, risks, and monitoring signals.
