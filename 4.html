What Equip means

Equip turns the working agent from Phase 3 into a capable agent by attaching tools, memory, permissions, runtime settings, and operational boundaries.

Capability provisioning

Name the tools, connectors, data sources, memory stores, and runtime permissions the agent needs.

Access boundaries

Define what each capability can read, write, call, spend, retain, modify, or escalate.

Operating safety

Add limits, approval gates, fallback behavior, and forbidden capability boundaries before Evaluate.

Research and recommend the capability map

Equip should recommend the first capability map, then ask the user to override only the parts that are wrong, risky, missing, or too broad.

Capability question Recommendation should cover Override prompt
What can the agent call? Tools, functions, MCP servers, connectors, APIs, browser, files, and databases. Accept these capabilities or remove tools the agent should not use.
What can the agent read or write? Data sources, write targets, scopes, credentials, and read-only boundaries. Raise or lower read/write scope, especially around production systems.
What memory does it use? Short-term state, long-term memory, retrieval sources, retention, and delete behavior. Change what is remembered, retrieved, retained, or forgotten.
What does the runtime need? Secret storage, environment variables, sandbox permissions, adapter settings, and deployment config. Change secrets, sandbox, adapter, or deployment assumptions.
What needs human approval? Actions that spend money, modify systems, send messages, publish, delete, or access sensitive data. Add approval gates for risky or sensitive capabilities.
What happens when a capability fails? Retry, fallback, degrade, ask, escalate, or stop behavior for each tool boundary. Change retry, fallback, escalate, or stop behavior.

Runtime config details

Recommend where each API key or credential should live for local development, CI/CD, production runtime, and managed operations.

Storage option Use when Equip recommendation
Local `.env`, `.env.local`, or runtime dev file The key is only needed for developer workstations or local sandbox runs. Keep it out of git, publish only `.env.example`, and avoid using production keys locally.
Platform runtime secrets The deployed agent runs on Cloudflare Workers/Pages, Vercel, Netlify, Firebase, or another managed runtime. Store production secrets in the host's secret system and inject them at runtime instead of baking them into code or static files.
CI/CD encrypted secrets The key is needed by GitHub Actions or another deploy pipeline. Use repository, environment, or organization secrets for build/deploy credentials, then pass runtime secrets into the hosting platform where possible.
Managed secret manager The agent needs audited, rotated, shared, or multi-environment secrets. Use a service such as Google Secret Manager, AWS Secrets Manager, Azure Key Vault, HashiCorp Vault, Doppler, Infisical, 1Password, or Bitwarden Secrets Manager.
Connector or OAuth token store The agent uses Gmail, Slack, Drive, GitHub, or another OAuth-backed connector. Prefer provider-managed connector storage or an encrypted backend token store rather than plain config files.
Browser-exposed public key The provider explicitly supports browser-side public keys, such as publishable keys or restricted map/browser keys. Restrict by domain, API, scope, quota, and environment; never expose server-side keys in client code.
Must cover

Scope by environment

Use separate dev, staging, and production keys so a test agent cannot mutate production systems by accident.

Must cover

Limit blast radius

Prefer read-only scopes, allowlists, referrer/IP restrictions, spend caps, and tool-specific approval gates.

Must cover

Plan rotation

Record who owns each secret, how it is revoked, how it is rotated, and what should happen when it leaks.

Operating limits

Equip is where access becomes bounded enough to test safely in Phase 5.

Allow

Approved tools, read scopes, runtime permissions, test data, and low-risk actions the agent may use alone.

Ask

Actions that need review, confirmation, elevated credentials, production writes, messages, or budget use.

Never

Forbidden tools, credentials, data classes, irreversible changes, or actions outside the agent's mandate.

Equip references

These references shape how MAPS thinks about tools, connectors, memory, dependencies, runtime settings, and observability handoff.

Tool and connector standard

Model Context Protocol

MCP is an open standard for connecting AI applications to external systems, including data sources, tools, and workflows.

Tools and guardrails

OpenAI Agents SDK

Reference for agents configured with instructions, tools, handoffs, guardrails, structured outputs, sessions, and tracing.

MCP connectors

OpenAI MCP and connectors

Reference for remote MCP servers and managed connectors that give models access to new capabilities.

Short-term memory

LangChain short-term memory

Reference for thread-scoped state that updates during agent invocations and tool calls.

Long-term memory

LangChain long-term memory

Reference for storing and recalling information across conversations, sessions, and namespaces.

Dependency injection

Pydantic AI dependencies

Reference for passing data and services to prompts, tools, and output validators in a type-safe way.

Runtime secrets

Cloudflare Workers secrets

Reference for production secrets that are passed to Workers like environment variables while remaining hidden after creation.

CI/CD secrets

GitHub Actions secrets

Reference for encrypted repository, environment, and organization secrets used by deploy workflows.

Managed secrets

Google Secret Manager

Reference for storing API keys, passwords, certificates, and other sensitive data in a managed secret store.

Managed secrets

AWS Secrets Manager

Reference for encrypted secret storage, retrieval, access control, and rotation for credentials and API keys.

Self-guided equip path

01

Read and research

Start from the working agent, then research comparable tools, connectors, memory patterns, and runtime settings.

Check it

The recommendation should name the patterns or references it is borrowing from.

02

Recommend access

Recommend tools, connectors, data access, memory, runtime permissions, credentials, and scopes.

Check it

Every recommendation should include reasoning, assumptions, risk, and confidence.

03

Recommend limits

Recommend approvals, budgets, rate limits, forbidden actions, and sandbox or production boundaries.

Check it

The agent should not be able to exceed its mandate by tool access alone.

04

Capture overrides

Ask the user to change only the recommendations that are wrong, risky, missing, or too broad.

Check it

User overrides should appear in `agents/{agent-handle}/capability-map.md`.
