Phase 2

Design

Design turns the agent brief into a buildable agent design. The default method is Research and Recommend: inspect the brief, research comparable agents, recommend the design choices, then capture user overrides.

Build navigation

Research and Recommend

Research and Recommend means the agent does the first pass of design judgment instead of handing the user a blank questionnaire.

Research

Find comparable agents, reference architectures, open-source implementations, product patterns, and design guidance.

Recommend

Recommend answers for operating model, roles, workflow, memory, tools, approvals, failures, and observability.

Override

Ask the user to accept the recommendations or override only the parts that are wrong, risky, or incomplete.

Research and recommend every design choice

The user should review a proposed design, not assemble one from scratch.

Design question Recommendation should cover Override prompt
Operating model Single agent, tool-using agent, supervised agent, or multi-agent participant. Accept this model or name the model you prefer.
Roles and responsibilities What the agent owns, what a human owns, and what other systems own. Remove, add, or rename responsibilities.
Workflow states Trigger, intake, plan, act, check, escalate, complete, and stop conditions. Change any state that does not match the real workflow.
Memory and context What is read, remembered, retrieved, retained, or intentionally forgotten. Tighten access, retention, or retrieval boundaries.
Tools and integrations Tool candidates, required permissions, failure behavior, and constraints. Approve, reject, or substitute tools.
Approvals and escalation Human approval gates, forbidden paths, escalation triggers, and owners. Raise or lower the authorization boundary.
Observability Logs, traces, review points, quality signals, cost, latency, and failure evidence. Add any signal needed to trust the design.
Test strategy How the agent will be proven before implementation: TDD, ATDD, scenario evals, or a blended approach. Change the proof style or raise the required evidence.
Acceptance scenarios Given/when/then examples that define correct behavior, edge cases, and escalation behavior. Add missing scenarios or reject weak proof.
Mock vs real tools Which tests use fakes, mocks, sandbox integrations, real APIs, or replayed traces. Change where real integrations are required.
Regression gates What must keep passing before the agent can move into Build, Evaluate, and Deploy. Add required gates before Build starts.
Backlog and dependencies Epics, build slices, task dependencies, priority order, first slice, deferred items, and proof required per item. Reorder, split, combine, defer, or mark blocked items before Build starts.
Runtime target and adapter requirement Primary runtime, secondary runtimes, skill format, tool interface, memory model, sandbox assumptions, packaging path, and portability risks. Choose the target runtime or mark the design runtime-neutral until Build.

Runtime target and adapter requirement

Design should decide where the agent may run. It should not build the adapter yet.

Runtime target

Name the primary runtime and any secondary runtimes, such as OpenClaw, Manus, Hermes, Codex, Claude Code, LangGraph, OpenAI Agents SDK, CrewAI, or Google ADK.

Profile or adapter

Decide whether the runtime only needs an install/use profile or a fuller adapter that transforms files, tools, memory, permissions, or packaging.

Portability risks

Record unsupported MAPS features, tool limitations, memory differences, sandbox constraints, approval gaps, and packaging risks before Build begins.

Backlog-driven build planning

Design should turn the approved direction into a dependency-aware backlog before Phase 3 starts.

Break it down

Split large projects into epics, slices, tasks, spikes, and deferred improvement items that can be reasoned about separately.

Map dependencies

Name prerequisites, blocked work, runtime adapter needs, data/setup work, and proof gates so Build does not start in the wrong order.

Prioritize

Order work by user value, risk reduction, dependency sequence, safety, and the smallest slice that proves real behavior.

Define the first slice

Choose the smallest vertical behavior that can be implemented, tested, reviewed, and handed to Equip without dragging the whole project behind it.

Defer consciously

Move polish, later capabilities, tech debt, and observation-driven improvements into the Phase 8 improvement backlog instead of hiding them in Build.

Create the artifact

Use `templates/agent-backlog.md` so the Build agent receives ordered work, dependencies, proof, and status.

Test-first proof plan

In Design, TDD is not implementation detail yet. It is a design requirement: define what evidence will prove the agent works before building it.

Test strategy

Recommend TDD, ATDD, scenario evals, integration tests, or a blended proof plan based on the agent's risk and tool use.

Acceptance scenarios

Write the given/when/then cases that define correct behavior, human escalation, refusal, recovery, and completion.

Eval shape

Decide whether proof uses deterministic tests, rubric-based evals, trace review, golden datasets, or human review gates.

Test balance

Recommend the unit, integration, and e2e mix before implementation choices make the system hard to test.

Mock policy

Name where mocks or fakes are acceptable, where sandbox tools are required, and where real integrations must prove behavior.

Build gate

State what must be proven before Phase 3 Build starts: failure cases, regression gates, and minimum acceptance evidence.

Adapter gate

State whether Build must create a runtime profile, a runtime adapter, or no adapter for the first working agent.

Self-guided design path

01

Read the brief

Start from `agents/{agent-handle}/agent-brief.md` and restate the goal, user, success criteria, and authority boundary.

Check it

The design should not invent a new job for the agent.

02

Research peers

Search for similar agents, tools, implementations, reference architectures, and product patterns.

Check it

The recommendation should name the comparable patterns it is borrowing from.

03

Recommend the design

Fill the recommendation table before asking for user decisions.

Check it

Every design question should have a recommendation and reasoning.

04

Capture runtime and overrides

Ask the user to change only the recommendations that do not fit, including runtime target and adapter requirements.

Check it

User overrides and runtime portability notes should appear in `agents/{agent-handle}/agent-design.md`.

05

Write the backlog

Turn the design into prioritized backlog items with dependencies, proof, first slice, and deferred improvement work.

Check it

Phase 3 should receive ordered work, not a vague project description.
