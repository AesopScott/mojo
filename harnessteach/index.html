Harness Skill Training

Harnesses are repeatable control wrappers around model calls, tool calls, workflows, and agent steps. They define the inputs, permissions, output shape, proof, and recovery behavior that make AI work controlled, tested, repeated, and trusted.

How to read this teacher page

Everything outside purple boxes is the student-facing spine. Everything in purple is for you: setup notes, scripts, warnings, model answers, and pacing. Purple means instructor-only.

The key teaching move: define harnesses positively as control wrappers around AI actions, then keep every example tied to the action being controlled.

Core definition

A harness is a repeatable control wrapper around AI work.

The important word is wrapper. The model, tool, or workflow is the thing doing work. The harness controls the conditions around that work.

Say it plainly

Harness = controlled wrapper

It controls what goes in, what is allowed to happen, what comes out, what proof is captured, and how the workflow recovers when the result needs another pass.

Study the pattern

Cards name harness skills

The harnesses board gives examples of control patterns: prompt control, tool control, proof control, routing control, memory control, and release control.

Development automation

Automation packages the harness

A harness becomes useful when it can be repeated as a skill, script, config, eval, workflow, guardrail, or release gate.

Orchestrator harness skills

How to read a card

Each card points to a possible harness skill.

Read a card by asking what it controls, how you would package it, and how you would know it worked.

Function

What control problem does it solve?

Example: Tool Registry controls which tools can run and under what permission rules.

Levers

What can be tuned?

Each card separates performance, cost efficiency, and capability as different design axes.

Skill package

What could become reusable?

A SKILL.md, script, policy file, eval suite, or workflow can package the control pattern.

Proof

How do you know it worked?

Look for logs, traces, evals, screenshots, review records, or state changes that prove behavior.

Vendor interfaces

You already use harnesses inside AI vendor tools.

Vendor dashboards and APIs expose small harness controls. The class is about recognizing those controls and learning how to build your own around higher-level workflows.

Vendor control Harness idea What it controls
Model picker Provider, Identity, Select, Route Which model is allowed to handle the task.
System or developer instructions Prompt, Policy, Goal The instruction boundary around the model call.
Temperature, max output, response format Prompt, Response, Cost, Latency How much variation, output shape, spend, and runtime are allowed.
Tool calling or function lists Tools, Tool Registry, Guardrails Which external actions the model may request.
File search, retrieval, context upload Knowledge Retrieval, Context, Memory Which information enters the model context.
Safety settings, filters, approval rules Safety, Privacy, Policy, Proof Authority What must be blocked, escalated, redacted, or reviewed.
Logs, traces, evals, usage dashboards Trace, Evidence, Verify, Trust How the run becomes inspectable after it happens.

Vendor interface script

How harnesses work

A harness skill turns judgment into a repeatable boundary.

If you can name the boundary, encode it, run it, observe it, and revise it, you are doing harness engineering.

1

Intent

What work is this AI step supposed to accomplish?

2

Inputs

What context, files, memory, and instructions are allowed in?

3

Authority

What model, tool, route, and action permissions are allowed?

4

Output

What shape, evidence, trace, and review state must come out?

5

Recovery

What happens when the result fails, stalls, costs too much, or needs review?

Build your own

Start small: build one harness skill around one risky step.

A useful harness skill starts with one clear boundary. Control that boundary well enough that the behavior can be repeated.

Build path

Seven decisions

  • Pick one AI action that needs control.
  • Describe what can go wrong.
  • Define allowed input, model route, tools, and output shape.
  • Choose proof: trace, eval, citation, screenshot, log, or review.
  • Define failure behavior: retry, fallback, block, ask, or escalate.
  • Package it as a SKILL.md, script, config, or workflow.
  • Test it with normal, edge, and unsafe examples.

Teacher-only

Build example to do live

Build a simple "source-citation harness" verbally. The controlled action is answering a factual question. The risk is unsupported claims. The harness requires sources, blocks low-confidence claims, logs the source list, and sends missing-evidence answers to review.

Keep it small. Students need to see that a harness can control one boundary clearly.

Teacher-only

Exercise facilitation

  • Give students 5 minutes to choose one card and fill the template.
  • Ask for one model-call example and one orchestration example.
  • Push every answer to include proof and failure behavior.
  • If the answer gets vague, redirect: "What AI action is being controlled?"

Exit ticket

Before you leave, answer this.

If you can answer these in plain language, you understand the core of harness engineering.

Exit ticket 
1. What AI action are you controlling?
2. What is allowed into the action?
3. What is allowed to happen during the action?
4. What must come out?
5. What proof makes the result trustworthy?
6. What happens when the harness blocks, fails, or needs review?
Teacher-only

Model answers

Prompt harness: controls system/developer instructions, allowed context, prompt template, output schema, and unsafe prompt patterns.

Tool Registry harness: controls tool allowlist, dry-run behavior, approval gates, rollback path, and tool-call logs.

Evidence harness: controls citation requirements, test records, review thresholds, trace links, and blocked trust states.

Final takeaway

Harness skills are how AI control becomes reusable engineering.

Vendor interfaces already give you small harness controls. Harness engineering is the practice of designing those controls deliberately, combining them into workflows, and packaging them so the same safety, proof, routing, and recovery logic can be reused.

Teacher-only

Closing line

Say: "A model call is easy to make. A trustworthy AI workflow is harder. Harness skills are the difference: they turn judgment, policy, proof, and recovery into reusable engineering."
