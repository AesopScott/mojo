Build navigation

What Observe means

Observe is the production feedback phase: instrument the deployed agent, monitor real behavior, review evidence, and turn what you learn into improvement work.

Runtime truth

Phase 7 looks at what the agent actually does after release, not what the eval suite hoped it would do.

Operational health

Observe tracks latency, cost, availability, queue depth, retries, rate limits, failed jobs, and stuck runs.

Improvement evidence

Phase 7 should produce evidence-backed observations that feed Phase 8, not dashboards that nobody acts on.

Phase 7 output

At the end of Observe, the required artifacts are `agents/{agent-handle}/observation-plan.md` and `agents/{agent-handle}/observation-log.md`.

Input

`agents/{agent-handle}/deployment-record.md`, production URL, release gate, smoke test evidence, eval report, capability map, and runtime logs/traces.

Skill

/observe-agent creates the observation plan, runtime signal map, incident criteria, review cadence, and improvement handoff.

Wrapper skills

/observe-agent+smith, /observe-agent+fuse, and /observe-agent+phoenix let students compare three Observe stacks against the same base workflow.

Templates

`templates/observation-plan.md` and `templates/observation-log.md` capture monitoring intent and real evidence.

Gate

The observation review should recommend continue, monitor closely, rollback, pause, or create Phase 8 improvement items.

Signals to observe

Phase 7 should make the most important production signals explicit before the agent disappears into normal operations.

Signal area What to watch Why it matters
Agent work Runs, traces, tool calls, connector calls, retries, approvals, fallbacks, final outputs, and escalation decisions. Shows whether the deployed agent is doing the intended loop.
Operating health Latency, cost, token use, uptime, queue depth, timeouts, rate limits, failed jobs, and stuck runs. Shows whether the agent is useful enough to operate repeatedly.
Quality User feedback, recurring failures, bad tool choices, incomplete tasks, eval drift, and regression patterns. Shows whether the agent remains helpful after release.
Safety and boundaries Denied tool calls, permission violations, approval gates, budget limits, unexpected data access, and unsafe requests. Shows whether the agent respects the capability map under real pressure.
Improvement evidence Incidents, repeated friction, eval misses, missing tools, slow handoffs, and support notes. Shows what should enter Phase 8 instead of becoming vague complaints.

Runtime evidence map

Observe should name where evidence lives and who reviews it.

Traces

Link LangSmith, Phoenix, Langfuse, OpenTelemetry, OpenAI Agents SDK, or runtime-specific traces to agent runs and eval baselines.

Logs and metrics

Name the production log source, dashboard, alert destination, latency/cost thresholds, error rates, and retention expectations.

Feedback

Capture user ratings, support tickets, Slack reports, annotation queues, escalation notes, and reviewer labels.

Ownership

Assign review cadence, owner, escalation path, rollback trigger, and Phase 8 backlog handoff responsibility.

Incident and review gates

Observation is useful only when the evidence can trigger action.

Continue

The agent is operating within expected quality, cost, latency, safety, and failure thresholds.

Monitor closely

The agent can stay live, but needs a narrower rollout, tighter budget, more review, or a tracked improvement item.

Rollback or pause

The agent violates safety boundaries, burns budget, fails the core loop, exposes data risk, or cannot be debugged from available evidence.

Improve handoff

If you cannot observe it, you cannot responsibly improve it.

Evidence

Write what happened

Record the trace, log, metric, feedback, incident, or support note that justifies the improvement.

Priority

Name the impact

Classify whether the observation affects safety, reliability, cost, latency, user value, model quality, or operator workload.

Backlog

Feed Phase 8

Convert the observation into `templates/improvement-backlog.md` with dependencies, priority, proof required, and target phase.

Self-guided observe path

01

Read the release

Start from the deployment record, eval report, capability map, production URL, and runtime-specific observability references.

Check it

The plan should inherit the release gate, rollback trigger, smoke checks, and known risk areas.

02

Recommend signals

Recommend traces, logs, metrics, feedback, alerts, thresholds, owners, review cadence, and incident triggers before asking broad questions.

Check it

Each signal should name where evidence lives and what action it can trigger.

03

Review evidence

Read production traces, metrics, logs, incidents, feedback, and eval drift, then record what happened.

Check it

The log should separate observed evidence from guesses and planned follow-up.

04

Feed Improve

Create or update Phase 8 improvement backlog items with evidence, priority, dependencies, proof required, and target phase.

Check it

Every improvement item should point back to a trace, log, metric, incident, eval failure, or user feedback signal.
