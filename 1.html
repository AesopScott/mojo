Phase 1

Define

Define turns an agent idea into a usable brief. The brief identifies the agent job, user or operator, desired outcome, authorization level, boundaries, success criteria, failure criteria, escalation points, risks, and assumptions. This is APS work inside the larger MAPS scaffold.

Agent brief fields

These are the minimum fields the next phase needs.

Name

Short identifier for the agent or role.

Handle and mandate

Stable agent handle plus the role or mandate it is expected to carry.

User or operator

The person or system that uses, reviews, or supervises the agent.

Job to be done

The specific outcome the agent is responsible for.

Persona and tone

Communication style for human-facing or agent-to-agent work.

Authorization level

What the agent may do alone, what needs approval, and what it must never do.

Tool access requested

Candidate tools, MCP servers, or capabilities the agent may need later.

Boundaries

In scope, out of scope, and unclear territory.

Success and failure

Observable criteria for done and explicit conditions that count as failure.

Risks and assumptions

What must be controlled, escalated, or proven later.

Define the authority boundary

Authorization is part of the agent definition. Do not wait until Design to decide what the agent can do.

01

Advise

The agent can analyze, recommend, summarize, or prepare options. A human decides and acts.

02

Draft

The agent can create messages, plans, code, tickets, or artifacts, but cannot publish or send them.

03

Act with approval

The agent can perform approved actions after a human confirms the specific action and target.

04

Act autonomously

The agent can act within explicit limits, logs its actions, and escalates when it leaves the defined lane.

!

Forbidden

Name the actions, data, systems, costs, and decisions the agent must never touch.

Always, ask first, never

Use this split when the authorization level is too vague.

Always allowed

Low-risk actions the agent can perform without asking, such as analysis, summarization, or local drafting.

Ask first

Actions that change external state, affect people, spend money, touch production data, or publish output.

Never allowed

Actions, data, systems, costs, and decisions that stay outside the agent's lane under every condition.

Self-guided define path

01

Name the job

Write the agent's job in one direct sentence with a clear owner and outcome.

Check it

If the sentence could describe several agents, it is still too vague.

02

Set authority

Choose the highest action level the agent is allowed to take and list what needs approval.

Check it

A reviewer should know exactly where the agent stops and where a human takes over.

03

Draw boundaries

Separate in-scope work, out-of-scope work, uncertain work, and forbidden actions.

Check it

The brief should prevent the most obvious misuse before any tools are connected.

04

Define proof

Write success criteria, failure criteria, escalation points, risks, and assumptions.

Check it

The next phase should be able to design evals from this definition.
