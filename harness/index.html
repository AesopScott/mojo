Harnesses for Model Control and Orchestration Control

A harness is the control layer around AI work. It decides what context enters, what model or worker runs, what tools are allowed, what evidence is captured, and when the result can be trusted.

Core idea

Harnesses make AI systems controllable enough to trust.

In this class, harnesses are not mainly about test runners. They are about controlling model calls and orchestrated work so an AI system can be routed, inspected, bounded, verified, and improved.

Keep this sentence

A harness wraps intelligence with control.

A model can produce an answer. An agent can take a step. A harness controls the conditions around that step: the goal, context, policy, route, tools, evidence, trust state, and recovery path.

Development automation is how those controls become repeatable: packaged as skills, scripts, gates, checks, evals, workflows, and installable project behavior.

Two control layers

Orchestration harnesses and model harnesses

The Harnesses page separates the control plane into two layers. One governs the system around the work. The other governs each model call.

Orchestration control

The system-level harness

This layer coordinates sessions, memory, tools, routing, proof, review, browser capture, Git control, and release readiness.

Model control

The per-call harness

This layer controls provider identity, selection, routing, prompt, context, policy, safety, privacy, tools, verification, trace, evidence, cost, latency, fallback, and trust.

The control loop

What a harness does around AI work

Use this loop when you inspect any card on the Harnesses page. The labels are different from classic software testing, but the trust logic is the same.

01

Bind intent

Connect the action to a goal, task, policy, or session.

02

Pack context

Choose what memory, files, rules, and examples enter the run.

03

Route work

Select the worker, model, tool, path, or fallback chain.

04

Constrain action

Apply safety, privacy, permission, approval, and cost boundaries.

05

Capture proof

Record traces, outputs, evidence, logs, screenshots, or reports.

06

Set trust state

Mark the result trusted, reviewed, blocked, expired, or needing fallback.

Class exercise

Pick one Harnesses-page card and translate it into a harness.

Choose a card from the live page. Use the worksheet to describe how it controls model work, orchestration work, or both.

Suggested cards

Good starting points

  • Proof Authority: how the system judges evidence.
  • Tool Registry: how tools become permissioned actions.
  • Model Router: how work is sent to the right model path.
  • Prompt: how admitted input is controlled.
  • Trace: how the route and output become auditable.
  • Trust: how the system decides what state the result is in.
Worksheet 
Card chosen:

What does this harness control?

Intent:
  What goal, policy, task, or session does it bind to?

Context:
  What information is allowed into the run?

Route:
  What model, worker, tool, or path does it choose?

Constraints:
  What safety, privacy, approval, cost, or authority limits apply?

Proof:
  What evidence, trace, output, or metric is captured?

Trust state:
  When is the result trusted, reviewed, blocked, expired, or retried?

Development automation:
  What skill, script, eval, gate, or workflow could package this harness?

The practical takeaway

A harness turns AI work from "the model said something" into controlled, inspectable, bounded, and repeatable system behavior.

Final line

Harness engineering is model control plus orchestration control, packaged into development automation.

That is why the Harnesses page matters: it shows the controls, the layers, the levers, the evidence model, and the skill packaging all in one place.
