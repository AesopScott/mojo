Global layer

Installed once. Available across projects.

Global installs are repos, plugins, hooks, vaults, and runtime helpers that sit above any one MAPS project. They should be documented separately from project skills so a developer can tell what is local to a repo and what is active across sessions.

This page explains the living inventory, but the MAPS source contract lives in catalogs/global-installs.md inside the MAPS+Org repo/plugin.

View global installs Open source catalog

Global contract

Global installs are not phase skills. They are shared capabilities a session may load, inherit, or call while running MAPS work.

Scope

Applies across projects, sessions, or tool surfaces instead of belonging to one project folder.

Disclosure

Document what is installed, what it changes, where it stores state, and when it activates.

Source contract

catalogs/global-installs.md is the MAPS plugin definition for global installs.

Separation

Keep global behavior distinct from MAPS phase skills so project scaffolds stay portable.

How MAPS should use this

Global installs should be considered during foundation, role, and skill-development work without being copied into every project by default.

M0 Foundation

Record whether a project uses a global memory or context layer, and whether that layer is canonical, mirrored, or read-only.

Skill behavior

Skills may reference global capabilities only when the current session has them installed or the project explicitly depends on them.

Project handoff

If a project depends on a global install, document the dependency so another developer knows what must be installed separately.

Adding more

Future global tools should be added here when they are installed outside a single project repo.

Required fields

Name, type, purpose, activation path, storage location, and whether it affects all sessions or only configured projects.

Examples

Global MCP servers, shared Obsidian vaults, global Codex skills, Claude Code hooks, repo templates, or cross-project runtime helpers.

Rule

If a global install changes session behavior, document it here before relying on it in a MAPS skill or classroom workflow.
