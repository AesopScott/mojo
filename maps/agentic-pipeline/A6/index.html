What Deploy means

Deploy turns a proved agent into a released agent by packaging it for the target runtime and recording exactly what shipped.

Release bridge

Phase 6 inherits the Phase 5 gate and moves only approved, conditional, or explicitly blocked work into release records.

Runtime-specific

Deployment may mean a Cloudflare Worker, Pages project, Python service, JavaScript package, OpenClaw profile, Hermes gateway, or Electron release.

Evidence-first

The deployment record must name the commit, build, runtime config, smoke test, release URL, and rollback path.

Phase 6 output

At the end of Deploy, the required artifacts are `agents/{agent-handle}/deploy-plan.md` and `agents/{agent-handle}/deployment-record.md`.

Input

`agents/{agent-handle}/eval-report.md` from Phase 5, plus the build artifact, capability map, runtime target, and release credentials.

Base skill

/deploy-agent creates the deploy plan, package notes, commands, evidence, and rollback plan.

Cloudflare wrapper

/deploy-agent++ extends base Deploy with GitHub Actions environments and Cloudflare deployment automation.

Templates

`templates/deploy-plan.md` and `templates/deployment-record.md` capture intent and evidence.

Release gate

The record should recommend deployed, package-only, specification-only, rolled back, or blocked.

Observe handoff

Phase 7 receives the production URL, runtime logs, trace links, incident triggers, and monitoring expectations.

Deploy coverage

The base skill should cover every deployment concern before any Cloudflare or GitHub wrapper adds platform-specific automation.

Deploy area What it proves Evidence to capture
Target runtime The release target is named and compatible with the evaluated agent. Runtime, environment, adapter/profile, account/project, and production URL.
Package artifact The thing being released is clear and reproducible. Artifact path, version, commit SHA, build ID, and checksum when available.
Manifest or metadata The runtime can understand how to install, load, or run the package. Manifest file, package metadata, entrypoint, commands, scopes, and required env vars.
Secrets and config Runtime secrets and CI/CD secrets are separated and scoped. Secret names, storage location, environment scope, owner, and rotation notes.
Preflight and smoke test The release was checked before and after deployment. Commands, results, release gate, smoke URL, and failure notes.
Rollback The team can recover from a bad release. Previous version, rollback command, owner, trigger, and post-rollback check.

The Deploy skills

Phase 6 uses a base skill for general deployment discipline and a separate `++` wrapper only for GitHub-to-Cloudflare automation.

Base

/deploy-agent

Use for any runtime. It covers target runtime, package artifact, manifest or package metadata, deploy commands, config handoff, preflight checks, smoke tests, evidence, rollback, and deployment record.

Wrapper

/deploy-agent++

Use when GitHub should deploy to Cloudflare. It adds GitHub Actions environments, Cloudflare Pages or Workers deploys, encrypted secrets, protection gates, smoke tests, and rollback evidence.

Boundary

Do not over-name

Only use `++` when the deployment actually adopts external capabilities. Plain packaging, manual commands, or runtime notes stay in the base skill.

Deployment modes

Deploy must be honest about whether the agent was actually released, merely packaged, or only specified for a future release.

Executable deploy

The agent can deploy now. Run the deployment path, smoke test the target, and record the live release evidence.

Package-only deploy

The deployable artifact can be built, but release is intentionally held. Record artifact details, commands, missing approvals, and the release procedure.

Specification deploy

The target runtime is not ready. Record exact commands, missing accounts, missing secrets, credentials, approvals, and manual steps. Do not claim deployment happened.

Release gates

Phase 6 ends with a release decision that can be audited later.

Deployed

The release reached the target runtime, passed smoke tests, and has rollback evidence.

Conditional

The release can proceed only with documented limits, manual approvals, narrow rollout, or monitoring conditions.

Blocked

The release is stopped because the eval gate, secrets, config, package, smoke test, or rollback plan is insufficient.

Self-guided deploy path

01

Read and research

Start from the eval report, capability map, target runtime, and comparable deploy references.

Check it

The recommendation should cite the runtime and any wrapper capability it is using.

02

Recommend release path

Recommend target runtime, artifact, metadata, deploy commands, secrets/config handoff, preflight, smoke test, and rollback before asking broad questions.

Check it

The plan should make GitHub deployment the default for Cloudflare projects when credentials and repo access are available.

03

Choose deployment mode

Use Executable deploy when the target can be released now, Package-only when the artifact is built but held, and Specification deploy when runtime pieces are missing.

Check it

The record should distinguish a live deployment from a package or future release specification.

04

Record and hand off

Capture release evidence, smoke test results, rollback path, and Observe handoff notes.

Check it

Phase 7 should inherit the production URL, trace/log links, monitoring signals, and rollback trigger.
