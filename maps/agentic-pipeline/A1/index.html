Define

Define turns an approved role into a usable agent brief. The skill now starts with the minimum source question, runs Research and Recommend from the role artifact and project context, then asks one accept, revise, or mark unknown question before it writes the brief and profile.

Agent layer

The agent brief is the contract.

A strong brief makes later design and evaluation possible because it states exactly what the agent owns. Define should not make the developer fill out the whole brief manually when the role artifact and research can support a recommendation.

Inputs

An existing role contract, role artifact, workflow need, or multi-agent role candidate. If no role exists yet, route back to /role.

Work

Research comparable agents and recommend scope, non-goals, authorization, observable success, failure criteria, profile settings, and escalation points.

Output

agents/{agent-handle}/agent-brief.md and agents/{agent-handle}/agent-profile.md, ready for Design and any org-chart profile surface.

Define uses Research and Recommend

The user should review a recommended agent definition, not assemble one from a blank form.

Ask the source first

If the source is missing, ask only: "Which role are we turning into an agent?" Do not ask the full interview list up front.

Research the role

Read the role contract, workflow, loop, state, project foundation, and comparable human or agent operating patterns.

Recommend the brief

Recommend the job, operator, outcome, voice, lifecycle status, autonomy, authority, tools, memory, handoffs, boundaries, proof scenarios, and profile controls.

Ask one confirmation

Ask whether the developer accepts the recommendation, wants to revise one part, or wants one item marked unknown.

Agent brief fields

These are the minimum fields the next phase needs.

Name

Short identifier for the agent or role.

Handle and mandate

Stable agent handle plus the role or mandate it is expected to carry.

User or operator

The person or system that uses, reviews, or supervises the agent.

Job to be done

The specific outcome the agent is responsible for.

Persona and tone

Communication style for human-facing or agent-to-agent work.

Authorization level

What the agent may do alone, what needs approval, and what it must never do.

Tool access requested

Candidate tools, MCP servers, or capabilities the agent may need later.

Profile controls

Activation status, autonomy level, authority level, granted/requested tool access, hook implementations, memory rights, and approval gates.

Boundaries

In scope, out of scope, and unclear territory.

Success and failure

Observable criteria for done and explicit conditions that count as failure.

Risks and assumptions

What must be controlled, escalated, or proven later.

Define the authority boundary

Authorization is part of the agent definition. Do not wait until Design to decide what the agent can do.

01

Advise

The agent can analyze, recommend, summarize, or prepare options. A human decides and acts.

02

Draft

The agent can create messages, plans, code, tickets, or artifacts, but cannot publish or send them.

03

Act with approval

The agent can perform approved actions after a human confirms the specific action and target.

04

Act autonomously

The agent can act within explicit limits, logs its actions, and escalates when it leaves the defined lane.

!

Forbidden

Name the actions, data, systems, costs, and decisions the agent must never touch.

Always, ask first, never

Use this split when the authorization level is too vague.

Always allowed

Low-risk actions the agent can perform without asking, such as analysis, summarization, or local drafting.

Ask first

Actions that change external state, affect people, spend money, touch production data, or publish output.

Never allowed

Actions, data, systems, costs, and decisions that stay outside the agent's lane under every condition.

Self-guided define path

01

Name the source role

Start with the role being turned into an agent. If the role does not exist yet, create the role first.

Check it

Define should not invent an agent from scratch when there is no role artifact.

02

Research and recommend

Use the role contract and comparable patterns to recommend the job, authority, memory, handoffs, and profile controls.

Check it

The developer should see a recommendation before being asked to fill missing fields.

03

Ask one confirmation

Ask whether the recommendation is accepted, one part should be revised, or one item should remain unknown.

Check it

No multi-question form, checklist, or table should be handed to the developer.

04

Write brief and profile

Create the brief and profile, then report completion, memory update, and the next skill.

Check it

The next phase should know whether to run /design-agent and what remains open.
