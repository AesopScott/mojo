Build

Build turns the approved agent design into a working agent. MAPS uses /build-agent++: base Build discipline, incremental implementation, and test-driven proof merged into one slice-by-slice workflow.

/build-agent++

The Phase 3 skill starts from the approved design, checks the Build gate, then implements one thin, testable behavior slice at a time.

Base Build

Keep the MAPS artifact contract: design in, working agent and build notes out.

Incremental implementation

Build vertical slices that leave the system working after every increment.

Test-driven proof

Write or update proof before behavior changes whenever the slice can be tested first.

Adapter/profile

If Design requires a target runtime, Build creates the smallest runtime adapter or runtime profile needed for the first working agent.

Gate

Build exits when the first working agent passes its defined checks, optional review is recorded, and the agent is ready for Phase 4 Equip.

Build from the backlog

Phase 3 should start from the dependency-aware backlog created in Design, not from a fresh interpretation of the project.

Check readiness

Confirm the selected backlog item has prerequisites, runtime setup, proof, and acceptance criteria before implementation starts.

Respect dependencies

Do not pull blocked items forward. Build prerequisite slices, spikes, adapters, or fixtures first.

Keep slices small

If an item is too large to prove independently, split it again and update the backlog before coding.

Update status

Record completed, blocked, deferred, and newly discovered backlog items in the build evidence.

Build one slice at a time

Each slice should add observable behavior, proof, and evidence without leaving the agent half-built.

Build question /build-agent++ should decide Evidence to capture
Which backlog item is ready? The highest-priority unblocked item from `agent-backlog.md`, or a smaller split item if the original is too large. Backlog ID, dependency check, priority, and ready/blocked status.
What is the first useful behavior? The smallest vertical slice that proves the agent can do real work. Scenario, test, trace, or runtime output.
What proof comes first? Unit test, integration test, scenario eval, scripted check, or manual verifier. Failing proof before implementation, then passing proof after.
What tool boundary is touched? Whether the slice uses a fake, sandbox, local tool, or real integration. Tool inputs, outputs, failure behavior, and permissions.
What adapter/profile is required? Whether the slice needs a runtime profile, transformed package, API wrapper, tool binding, or memory setup. Adapter files, runtime assumptions, verification command, and known incompatibilities.
What can be safely deferred? Nonessential polish, broad abstractions, or Phase 4 capabilities. Deferred notes and reason.
What must keep passing? Regression checks and acceptance gates from Phase 2. Command output, trace, screenshot, or review note.

Runtime adapter build

Build creates runtime-specific packaging only when the first working agent needs it.

Runtime profile

Use a lightweight profile when the runtime can consume MAPS-style skills directly and only needs install paths, repo instructions, tool discovery, or usage notes.

Runtime adapter

Build an adapter when the runtime needs transformed files, API wrappers, tool bindings, memory setup, permission gates, or import/export packaging.

Runtime proof

Verify the first slice in the selected runtime or a safe sandbox, then record known incompatibilities for Equip, Evaluate, and Deploy.

Optional code review

Phase 3 should raise code review as an optional Build gate before Equip, especially when the agent touches tools, credentials, runtime adapters, production code, or user-facing behavior.

Standard review

Run a code review pass for bugs, security issues, missing tests, broken assumptions, runtime edge cases, and handoff gaps.

Different-model review

For higher-risk builds, run a second review with a different model or reviewer to catch blind spots from the builder model.

Review evidence

Record whether review was skipped, completed, or escalated, plus findings, fixes, unresolved risks, and why a second pass was or was not used.

Self-guided build path

01

Check the gate

Read the Phase 2 design and backlog, then confirm the Build gate is satisfied.

Check it

The build should not start from unresolved design questions.

02

Pick runtime and backlog item

Choose the target runtime/profile and the highest-priority unblocked backlog item that can be proven as a small slice.

Check it

The slice should be observable through a test, eval, trace, or runtime check in the selected runtime or sandbox.

03

Prove then build

Write the proof, watch it fail for the right reason, implement, then verify.

Check it

If test-first is not possible, document why and add the closest guard.

04

Log evidence

Record checks, decisions, blockers, and handoff notes for Equip and Evaluate.

Check it

The next phase should not need to rediscover what was built or proven.
