Org Chart

Roles versus Agents

Roles describe responsibility, voice, maturity, and authority. Agents are implemented systems with state, tools, memory, policies, evaluations, handoffs, and stop conditions.

Next: role maturity Back to org chart

Why the distinction matters

A role describes organizational responsibility. An agent is an implemented operating system for that role. MAPS keeps those separate so a title never accidentally grants tools, memory writes, production access, or autonomy.

Role

A role is a named organizational seat with purpose, voice, responsibilities, maturity, authority boundaries, and evidence expectations.

Role agent candidate

A role becomes a candidate when it has a contract and may be useful as an advisor, review gate, workflow owner, operator, or active loop.

Implemented agent

An agent has a goal, state, policy, tools, memory, evaluations, handoffs, escalation path, stop conditions, and an approved build path.

Decision table

Use this table before turning any organizational role into software.

QuestionRole answerAgent answer
What is it?A responsibility and authority contract.A running implementation that can perform work.
How does it speak?In first person as the role.In first person as the role, constrained by runtime policy.
What can it do?Only what its declared authority allows.Only what its tools, memory, policy, approvals, and stop conditions allow.
When does it activate?When invoked or when its engagement rule matches the work.When triggered by explicit call, event, schedule, hook, queue, or workflow state.
What proves maturity?Useful judgment, repeatable outputs, review evidence, and trusted scope.Passing evaluations, reliable traces, safe execution, and monitored operations.

MAPS rule

Creating a role contract is not the same as building an agent. Agent build status requires explicit approval and a build-ready design.
