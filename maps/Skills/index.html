MAPS skills

The rules that make MAPS repeatable.

MAPS skills are installable Codex workflows. Each skill asks one question at a time, produces a named artifact, records completion, writes helper notes, and points the builder to the next skill.

This page explains the rules behind the skills and where those rules are implemented in the downloadable skill files, templates, helper scripts, and validator.

Review the rules Open skill files

Skill rules

Every MAPS skill follows the same operating contract so a new session can pick up the method without remembering this class conversation.

One question at a time

Skills interview the builder sequentially. They do not dump a form of unrelated questions into the chat.

Research and recommend

After the essential user inputs are known, skills should research, compare options, and recommend a path instead of making the user design every detail from scratch.

Explicit output

Each skill states the concrete file, plan, role, scaffold, or implementation artifact it is responsible for producing.

Completion report

Every skill ends by saying whether it is complete, what changed, what memory was updated, and which skill should run next.

Versioned behavior

Each skill carries a version and changelog so rule changes are visible when the skill evolves.

Source rule catalog

catalogs/skill-rules.md defines the MAPS plugin rules and where each rule is implemented.

Memory-aware execution

Project skills write through the shared helper so notes, RAG mirrors, and the run log stay aligned.

Role engagement

Role skills separate how a role participates from what it is allowed to do: passive reference, advisory, review gate, workflow owner, operator, autonomous loop, or escalation authority.

First-person roles

Role agents speak as the role in first person, with a defined voice, point of view, activation marker, and clear limits. They do not answer as Claude, Codex, ChatGPT, or an outside narrator.

How they run

A MAPS skill is a local folder with a `SKILL.md` file and optional scripts, templates, and references.

Layer Implementation Purpose
SKILL.md Frontmatter, workflow, versioning, output contract, completion report. Teaches Codex how to perform the phase or role workflow.
templates/ Phase artifacts such as project-foundation.md and system-shape.md. Gives the skill a concrete deliverable shape.
references/ Research sources, role taxonomies, repo notes, and domain guidance. Lets a skill load deeper context only when needed.
scripts/ Deterministic helpers such as memory routing and skill validation. Keeps repeated behavior reliable instead of retyping it into prompts.

Recent rule changes

These changes were added to the installed skills, the Mindshare project copies, and the downloadable MAPS training assets.

Define Agent v0.5.0

/define-agent now asks only for the source role when missing, then researches and recommends the brief, authority, memory, proof, and profile controls before asking one confirmation question.

Design Agent v0.3.0

/design-agent now asks only for the source brief or role contract when missing, then recommends workflow, voice, memory, tools, approvals, proof, risks, and handoffs before asking one confirmation question.

Design is a build gate

Role-to-agent design creates the design contract, runtime contract, build backlog, proof plan, and build gate. Full implementation still belongs to /build-agent, with Equip and Evaluate evidence required before operational confidence.

Build Agent v0.7.0

/build-agent now requires Research and Recommendation before each Autonomy Contract Interview question. Build must draft a recommended answer from source artifacts, keep single-role promotion scoped to that role, and ask Scott to approve or revise before implementation.

Evaluate proof mode

/evaluate-agent must distinguish specification-mode proof from executable proof. A specification-mode pass can confirm profile and design readiness, but implemented Agent status still requires Build, Equip, and Evaluate evidence for runnable loop, tool wiring, and runtime behavior.

Role voice

Generated roles speak in first person as the role. They should not preface answers with narrator language such as "Before I answer as..." or "Speaking as..."

Role lifecycle

The lifecycle states are now unauthorized, authorized role, authorized agent, suspended, and retired. Maturity, lifecycle, authority, and org-chart color are separate concepts.

Role memory

/role now includes a per-role memory template so each role can learn and grow through a named memory file instead of relying only on the role contract.

Role heartbeat template

/role v0.17.0 adds templates/heartbeat-automation.md so new roles inherit the 5-minute handoff check pattern, split channel map, checked-location reporting, quiet no-work behavior, and repo-local memory rules.

Memory record

The Define and Design corrections were recorded through MAPS helper notes and in ObsidianMind so later sessions can see why the rules changed.

Memory writes

The shared helper is a script that runs when a skill completes. It is not a background service.

Control file

project-foundation.md defines the memory contract, RAG/notes locations, run log, and scaffold decisions.

Helper notes

Phase skill notes in maps-runs use [project]-[skill]-helper-notes.md. Role notes use role-[role-name].md.

Reindex manifest

.maps/rag-updates.json records changed notes and marks them for future RAG reindexing.

Validation

The skill contract validator checks that published MAPS skills carry the rules developers use in real projects.

Validator check Why it matters
Version and changelog Rule changes become trackable instead of disappearing into a session.
Output contract Developers know what the skill is expected to create.
One-question interview Skills gather information conversationally and reduce confusion.
Completion report Each run closes with status, outcome, memory update, and next skill.
Proof mode boundary Specification-mode evaluation cannot be mistaken for implemented Agent runtime confidence.
Build 10/10 proof Role-to-agent builds need runnable command evidence, fail-closed contract tests, conflict tests, denied-behavior tests, audit artifacts, runtime-adapter proof timing, and Equip/Evaluate handoffs.
Autonomy Contract Gate Autonomous behavior requires a named goal, authority domains, tools, memory/state rights, trigger or cadence, approval gates, stop conditions, eval proof, audit logs, rollback path, and owner before Build may implement it.
Memory helper wiring Phase artifacts, helper notes, and RAG update manifests stay synchronized.
Role engagement taxonomy Role contracts define participation style separately from authority and implementation.
First-person role voice Role contracts define how the role speaks as itself, including prohibited narrator language.
