01 // TOPIC DETAIL

Roles vs. Agents

  1. Do not start by asking, “What agent should I build?”
  2. Start by asking, “What work needs to be done?”
  3. That work may be described as a task, role, function, workflow, process, job, responsibility, or capability.
  4. The label matters less than the boundary.
  5. Before building an agent, define the unit of work.
  6. A unit of work needs a purpose.
  7. It needs inputs.
  8. It needs outputs.
  9. It needs success criteria.
  10. It needs authority limits.
  11. It needs escalation rules.
  12. It needs a human owner.
  13. Only then should you decide whether that work should be handled by a prompt, a tool, a workflow, an assistant, an agent, or a multi-agent system.
  14. An agent is not the work itself.
  15. It is one possible execution mechanism for the work.
  16. The mistake is turning every useful task into “an agent” before you understand the task.
02 // TOPIC DETAIL

Agent Contracts

  1. A contract defines the agreement between the work and the worker.
  2. It says what the agent is expected to do and what it is not allowed to do.
  3. A contract can apply to a task, workflow, role, capability, assistant, agent, or service.
  4. The point is not the label. The point is explicit expectations.
  5. A contract starts with purpose. Why does this unit of work exist?
  6. A contract defines inputs. What information, files, messages, events, tickets, prompts, or data does the agent need?
  7. A contract defines outputs. What should the agent produce: draft, summary, decision support, ticket update, report, code, recommendation, or handoff?
  8. A contract defines authority. What can the agent decide or do on its own?
  9. A contract defines boundaries. What must the agent never do, even if it technically can?
  10. A contract defines escalation rules. When should the agent stop and ask a human, supervisor, or another system?
  11. A contract defines success criteria. How do we know the work was done well?
  12. A contract defines failure conditions. What does “bad output,” “unsafe action,” “low confidence,” or “incomplete work” look like?
  13. A contract defines tool permissions. Which tools, APIs, files, databases, browsers, or systems may the agent access?
  14. A contract defines data rules. What data can be used, stored, shared, summarized, retrieved, or excluded?
  15. A contract defines communication rules. How should the agent hand off work, cite sources, report blockers, express confidence, and preserve context?
  16. A contract defines review and approval points. What requires human review before it becomes official, external, published, sent, deployed, or paid for?
  17. A contract turns “the AI should help” into an operating agreement.
  18. Without a contract, you have vibes and tool access. With a contract, you have accountable automation.
03 // TOPIC DETAIL

Orchestration Patterns

  1. Orchestration is how work moves through the system.
  2. It defines who starts, who decides, who acts, who reviews, and where the work goes next.
  3. The simplest pattern is single-agent execution. One agent receives a task, uses its tools, produces an output, and stops.
  4. A supervisor pattern uses one coordinator to route work. The supervisor decides which agent or tool should handle each part of the task.
  5. A planner-executor pattern separates strategy from action. One agent creates the plan; another agent or set of agents performs the steps.
  6. A router pattern classifies the request and sends it to the right worker. This is useful when different tasks need different skills, tools, risks, or approval paths.
  7. A peer-to-peer handoff pattern lets agents pass work directly. Each agent completes its part, packages the state, and hands it to the next agent.
  8. A queue-based pattern turns work into durable tasks. Agents pick up jobs from a queue, process them, update status, and leave an audit trail.
  9. A critic/reviewer pattern separates creation from evaluation. One agent drafts; another checks for correctness, safety, quality, or policy fit.
  10. A debate pattern uses multiple agents to test competing interpretations. This is useful for ambiguous decisions, but it can become noisy if not bounded.
  11. A consensus pattern requires agreement before action. It can improve confidence, but it adds latency and should be reserved for higher-risk work.
  12. A human-in-the-loop pattern places approval gates at key points. Humans approve external messages, production changes, spending, legal/medical/HR decisions, or anything with real-world consequence.
  13. A tool-calling pattern lets agents use external systems. The orchestration must define which tools are available, when they can be used, and what requires approval.
  14. A memory-augmented pattern injects context before action. The system retrieves relevant notes, history, documents, policies, or prior decisions so the agent is not operating from scratch.
  15. A fallback pattern defines what happens when confidence is low or a tool fails. The agent may retry, escalate, ask for clarification, switch tools, or stop safely.
  16. A good orchestration pattern is chosen by risk, complexity, and handoff needs.
  17. Do not use five agents where one tool call is enough. Use more structure only when the work requires coordination, review, memory, or authority boundaries.
04 // TOPIC DETAIL

Communication Protocols

  1. A communication protocol defines how work is passed between agents, tools, systems, and humans.
  2. It prevents handoffs from becoming vague conversation.
  3. A good protocol makes every message actionable.
  4. The receiver should know what happened, what is needed, what is blocked, and what to do next.
  5. Every handoff should include task identity. What task, ticket, request, workflow, customer, file, or project is this about?
  6. Every handoff should include source context. What information was used, where did it come from, and what should the next worker trust or verify?
  7. Every handoff should include current state. Is the work new, in progress, blocked, waiting for review, approved, rejected, or complete?
  8. Every handoff should include the requested next action. Summarize, draft, review, approve, execute, investigate, escalate, test, publish, deploy, or stop.
  9. Every handoff should include assumptions. Agents should say what they are treating as true, especially when the source material is incomplete.
  10. Every handoff should include confidence. The system should distinguish between “verified,” “likely,” “uncertain,” and “needs human review.”
  11. Every handoff should include blockers. Missing data, failed tools, unclear authority, policy conflicts, or low confidence should be explicit.
  12. Every handoff should include boundaries. The receiver needs to know what actions are allowed, prohibited, or approval-gated.
  13. Communication should be structured when reliability matters. Use fields like task_id, source, status, next_action, confidence, blockers, and approval_required.
  14. Communication can be conversational when exploration matters. Brainstorming, coaching, creative work, and early discovery may need looser exchange before structure is useful.
  15. Protocols should separate facts from recommendations. Facts say what is known. Recommendations say what the agent thinks should happen.
  16. Protocols should preserve traceability. A later reviewer should be able to tell who did what, why, from what source, and under what authority.
  17. The goal is not more messages. The goal is cleaner handoffs.
  18. A good communication protocol reduces ambiguity, repeated work, unsafe assumptions, and lost context.
05 // TOPIC DETAIL

Shared Memory and Context Injection

  1. Shared memory is what the system can remember across tasks.
  2. Context injection is how the right memory gets brought into the current task.
  3. Memory and context are not the same thing.
  4. Memory is stored information. Context is the selected information given to the agent right now.
  5. Good systems do not inject everything.
  6. They inject the smallest useful set of notes, facts, files, policies, examples, and decisions.
  7. Shared memory should have ownership.
  8. Someone or some process must decide what gets stored, updated, trusted, deprecated, or deleted.
  9. Memory should be source-aware.
  10. Agents need to know where information came from: user input, policy, database, transcript, ticket, document, code, or prior agent output.
  11. Memory should have trust levels.
  12. A signed policy, a draft note, an old chat summary, and an unverified model output should not carry the same weight.
  13. Memory should be scoped.
  14. Personal memory, project memory, customer memory, role memory, organization memory, and system memory should not be treated as one giant pile.
  15. Context injection should match the task.
  16. A writing agent needs different context than a release agent, support agent, security agent, or coding agent.
  17. Context injection should include constraints, not just facts.
  18. Policies, boundaries, approval rules, data rules, and failure conditions are often more important than background information.
  19. Memory can improve continuity, but it can also preserve mistakes.
  20. Bad memory creates confident repetition of old errors.
  21. Agents should say what memory they used.
  22. This makes outputs easier to audit and helps humans catch stale or irrelevant context.
  23. Memory needs update rules.
  24. Define when an agent may propose a memory update, when a human must approve it, and when a memory should expire.
  25. Retrieval quality matters more than storage volume.
  26. A huge memory store is not useful if the wrong context is injected at the wrong time.
  27. Shared memory requires privacy and access control.
  28. Not every agent should see every note, customer record, internal decision, secret, or personal detail.
  29. The goal is not to make agents remember everything.
  30. The goal is to make them remember the right thing at the right moment, with the right boundaries.
06 // TOPIC DETAIL

Tool Use and Capability Boundaries

  1. Tool use is what lets an agent affect the world outside the chat.
  2. Reading a file, calling an API, sending an email, updating a ticket, running code, or changing a database are tool actions.
  3. A capability is broader than a tool.
  4. A tool is the mechanism. A capability is what the agent can actually accomplish with it.
  5. Tool access should follow the work, not curiosity.
  6. Do not give an agent a browser, email, database, shell, or CRM just because it might be useful.
  7. Every tool should have a reason.
  8. The contract should explain why the agent needs the tool and what work the tool supports.
  9. Every tool should have a boundary.
  10. Define what the agent may read, write, create, edit, send, delete, purchase, approve, or execute.
  11. Read access and write access are different risks.
  12. Reading a calendar is not the same as scheduling a meeting. Reading a draft is not the same as sending it.
  13. External actions need stronger controls.
  14. Sending emails, posting content, submitting forms, messaging customers, publishing pages, or changing cloud systems should usually require approval.
  15. Production tools should be gated.
  16. Deployments, database writes, infrastructure changes, access-control changes, and security settings need explicit authority and audit trails.
  17. Secrets should not be casually exposed to agents.
  18. API keys, passwords, tokens, private credentials, and signing keys require strict handling or should stay outside agent reach.
  19. Agents should use least privilege.
  20. Give the smallest set of tools, scopes, records, and actions needed to complete the work.
  21. Tool calls should be observable.
  22. Log what tool was used, with what input, for what task, and what result came back.
  23. Tool failures need defined behavior.
  24. The agent should know whether to retry, switch tools, ask for help, escalate, or stop.
  25. Tool outputs are not automatically truth.
  26. Search results, API responses, scraped pages, OCR, and database records may be stale, partial, wrong, or misinterpreted.
  27. Capability boundaries should be tested, not assumed.
  28. Try scenarios where the agent is asked to overreach: send without approval, access restricted data, delete records, or bypass policy.
  29. The goal is useful power under control.
  30. An agent with no tools may be harmless but limited. An agent with too many tools may be productive and dangerous. The engineering problem is matching capability to authority.
07 // TOPIC DETAIL

Governance and Human Approval Gates

  1. Governance defines what the system is allowed to do.
  2. It turns broad AI capability into controlled operating behavior.
  3. Human approval gates define where automation must stop.
  4. They mark the points where an agent, workflow, or tool must wait for a person before continuing.
  5. Approval gates should match consequence, not complexity.
  6. A simple action can still need approval if it affects customers, money, production, compliance, or reputation.
  7. External communication usually needs a gate.
  8. Emails, customer messages, social posts, legal notices, public claims, and partner communications should not be sent without the right approval.
  9. Production changes need explicit approval.
  10. Deployments, database writes, infrastructure changes, security settings, and access-control changes should be gated.
  11. Spending and financial actions need approval.
  12. Purchases, renewals, refunds, invoices, pricing changes, and financial commitments should not be delegated casually.
  13. Sensitive data requires special handling.
  14. Medical, legal, HR, financial, customer, security, and personal data should have clear access, use, retention, and sharing rules.
  15. Agents should know when to stop.
  16. A good system defines stop conditions for uncertainty, policy conflict, missing authority, high-risk data, or failed validation.
  17. Governance should define who can approve what.
  18. Approval is not generic. Different decisions may require a manager, legal reviewer, security owner, clinician, release manager, or executive.
  19. Approval gates should be visible in the workflow.
  20. Humans should be able to see what is waiting, why it is waiting, who owns it, and what decision is needed.
  21. Approvals should preserve context.
  22. A human approver needs the recommendation, supporting facts, source material, risks, assumptions, and proposed next action.
  23. Governance should be testable.
  24. The system should be tested against overreach: sending without approval, accessing restricted data, making claims, spending money, or changing production.
  25. Governance should include audit trails.
  26. Record who approved, what was approved, when it happened, what evidence was used, and what action followed.
  27. Human approval should not become meaningless rubber-stamping.
  28. If every tiny action requires approval, people stop reviewing carefully. Gate the actions that actually carry risk.
  29. The goal is not to slow the system down.
  30. The goal is to let useful automation move quickly while keeping consequential decisions accountable.
08 // TOPIC DETAIL

Testing, Evaluation, and Observability

  1. Testing asks whether the system works.
  2. It checks expected behavior against known scenarios.
  3. Evaluation asks whether the system is good enough.
  4. It measures quality, usefulness, safety, correctness, and consistency.
  5. Observability asks what actually happened.
  6. It gives you traces, logs, metrics, costs, failures, and decisions after the system runs.
  7. Test scenarios should reflect real work.
  8. Use actual task types, edge cases, bad inputs, missing information, conflicting instructions, and high-risk requests.
  9. Each agent needs unit-style tests.
  10. Test whether the agent follows its contract, respects boundaries, produces the right output shape, and escalates when required.
  11. The full workflow needs end-to-end tests.
  12. Multi-agent systems fail at handoffs, routing, memory injection, tool calls, review gates, and state transitions.
  13. Evaluation needs a rubric.
  14. Define what good means: accurate, complete, sourced, concise, safe, useful, policy-compliant, and appropriate for the audience.
  15. Outputs should be evaluated separately from actions.
  16. A good draft and a safe tool action are different quality problems.
  17. Trace every important step.
  18. Record which agent acted, what context it received, what tool it used, what it produced, and what happened next.
  19. Measure failure modes directly.
  20. Track hallucinations, missed escalations, bad routing, stale memory, tool failures, policy violations, and low-confidence outputs.
  21. Watch cost and latency.
  22. A system that is accurate but too slow or expensive may still fail operationally.
  23. Use regression tests.
  24. When you fix a failure, add it to the test set so the system does not quietly relearn the same mistake.
  25. Human feedback should become evaluation data.
  26. Approvals, rejections, edits, complaints, and corrections are signals the system can learn from.
  27. Observability should support accountability.
  28. A reviewer should be able to reconstruct what happened, why it happened, and who approved the consequential steps.
  29. The goal is not to prove the system is perfect.
  30. The goal is to know where it is reliable, where it is risky, and when it needs human judgment.
09 // TOPIC DETAIL

Building a Real Multi-Agent Workflow

  1. Start with the work, not the agents.
  2. Define the actual business process before naming workers or tools.
  3. Choose one concrete workflow.
  4. Use something specific: intake a customer request, draft a proposal, review a pull request, triage an incident, create training content, or prepare a release.
  5. Define the trigger.
  6. What starts the workflow: a form submission, email, ticket, Slack message, file change, scheduled event, or human request?
  7. Define the first state.
  8. The system needs to know what “new work” looks like before anyone touches it.
  9. Break the workflow into stages.
  10. Common stages are intake, classification, research, planning, execution, review, approval, delivery, and follow-up.
  11. Assign each stage to a worker.
  12. The worker may be a human, prompt, tool, script, workflow, assistant, agent, or another system.
  13. Write a contract for each worker.
  14. Define purpose, inputs, outputs, tools, authority, boundaries, escalation rules, and success criteria.
  15. Define the handoff packet.
  16. Each stage should pass forward task ID, source context, current state, output, assumptions, blockers, confidence, and next action.
  17. Add memory and context injection deliberately.
  18. Decide what project notes, policies, examples, prior decisions, or customer history should be injected at each stage.
  19. Add tool access only where needed.
  20. Do not give every worker every tool. Match tools to the stage and authority of the worker.
  21. Place human approval gates before consequential actions.
  22. Gate external communication, production changes, spending, sensitive data use, public claims, legal/medical/HR decisions, and final release.
  23. Define failure and fallback behavior.
  24. If a worker is uncertain, blocked, missing data, or sees policy risk, it should know whether to retry, escalate, ask for clarification, or stop.
  25. Track state across the workflow.
  26. Use statuses like new, assigned, in progress, blocked, waiting for review, approved, rejected, delivered, and complete.
  27. Test the workflow end to end.
  28. Run normal cases, edge cases, tool failures, bad inputs, missing approvals, and attempts to bypass boundaries.
  29. Observe and improve the workflow after it runs.
  30. Review traces, handoffs, cost, latency, failures, edits, approvals, and human feedback so the system gets more reliable over time.
Multi-Agent System Primer

Comprehensive Build Primer

A multi-agent system is not built by naming agents first. It is built by defining the work, dividing responsibility, writing contracts, controlling tools, injecting the right context, placing approval gates, and testing the system until the handoffs are reliable.

Purpose

Build a website pop-up or modal experience with nine collapsible steps. Each step should help a user walk through the process of designing a multi-agent system.

The user should not be asked to start by naming agents. The primer should guide them to define the work, break it into stages, assign responsibilities, write contracts, design handoffs, define memory/context, assign tools, add governance gates, and test the system.

Step 1: Define The Work

Start with the business process, task, workflow, or outcome. Do not start by naming agents.

Ask AI 
Help me define the work before I build agents. Ask me what business process, task, workflow, or outcome I want to improve. Then produce a 15-item work-definition checklist that answers:
1. What work needs to be done?
2. What business problem does it solve?
3. Who benefits from the work?
4. Who owns the outcome?
5. What starts the work?
6. What is the expected result?
7. What inputs are required?
8. What outputs should be produced?
9. What decisions must be made?
10. What can be automated safely?
11. What should remain human-owned?
12. What risks or constraints exist?
13. What would make this work successful?
14. What would make this work fail?
15. Why might this need an agentic system instead of a simple prompt, script, or automation?

Successful Output: A clear description of the work, not yet a list of agents.

Step 2: Break The Work Into Stages

Turn the work into a practical sequence of stages.

Ask AI 
Break this workflow into clear stages. For the full workflow, produce a 15-item stage map that answers:
1. What is the first stage?
2. What is the final stage?
3. What happens in each stage?
4. What information is needed at each stage?
5. What output is produced at each stage?
6. What decision happens at each stage?
7. What handoff happens after each stage?
8. Which stages are simple enough for a prompt or tool?
9. Which stages may need an agent?
10. Which stages need human ownership?
11. Which stages need human review?
12. Where can the workflow get blocked?
13. Where can the workflow branch?
14. Where should state be tracked?
15. Which stages are unnecessary or should be combined?

Successful Output: A staged workflow such as intake, classify, research, plan, execute, review, approve, deliver, and follow up.

Step 3: Identify Workers, Roles, Or Capabilities

Decide what kind of worker belongs at each stage. A worker may be human, AI, scripted, tool-based, or external.

Ask AI 
For each stage of this workflow, identify what kind of worker is needed. The worker may be a human, prompt, tool, script, workflow, assistant, agent, or external system. Produce a 15-item worker map that answers:
1. What work must be done at this stage?
2. What kind of worker fits the work?
3. Why is that worker needed?
4. Could a simple prompt handle it?
5. Could a script or automation handle it?
6. Does it need an agent?
7. Does it need multiple agents?
8. Does it need a human owner?
9. Does it need human review?
10. What capability does the worker need?
11. What domain knowledge does the worker need?
12. What tools might the worker need?
13. What should the worker not be allowed to do?
14. What output should the worker produce?
15. How does this worker hand off to the next stage?

Successful Output: A participant map that does not turn everything into an agent by default.

Step 4: Write The Contract For Each Worker

Define responsibility, authority, boundaries, and failure behavior before implementation.

Ask AI 
Create an operating contract for each worker in this workflow. For each worker, answer these 15 contract questions:
1. What is the worker's purpose?
2. What task, role, workflow, or capability does it support?
3. What inputs does it require?
4. What outputs must it produce?
5. What tools may it use?
6. What data may it access?
7. What authority does it have?
8. What actions are prohibited?
9. What requires human approval?
10. When must it escalate?
11. What does successful work look like?
12. What does failure look like?
13. What communication format should it use?
14. What memory or context may it use?
15. Who owns or reviews its output?

Successful Output: Clear contracts that define responsibility and limits before implementation.

Step 5: Design The Handoff Protocol

Specify how work moves from one stage or worker to the next.

Ask AI 
Design the handoff protocol for this multi-agent workflow. Produce a 15-field handoff packet that includes:
1. Task ID
2. Workflow name
3. Source or trigger
4. Current stage
5. Current status
6. Worker sending the handoff
7. Worker receiving the handoff
8. Source context used
9. Output produced
10. Assumptions made
11. Confidence level
12. Blockers or missing information
13. Boundaries or prohibited actions
14. Approval requirements
15. Requested next action

Successful Output: A reusable handoff packet format.

Step 6: Define Memory And Context Injection

Decide what each stage needs to know and what should stay out of context.

Ask AI 
For each stage of this workflow, define what memory and context should be injected. Produce a 15-item context-injection plan that answers:
1. What context does this stage need?
2. What memory source should provide it?
3. Who owns that memory?
4. How trustworthy is the source?
5. Is the context personal, project, customer, role, organization, or system memory?
6. What policies or constraints must be injected?
7. What examples should be injected?
8. What prior decisions matter?
9. What source documents matter?
10. What should not be injected?
11. What privacy or security limits apply?
12. How much context is enough?
13. How should the worker cite or report memory use?
14. When should memory be updated?
15. When should memory expire or be reviewed?

Successful Output: A context map that says what each worker needs to know and what it should not see.

Step 7: Assign Tools And Capability Boundaries

Give each worker only the tools and permissions required for its stage.

Ask AI 
Map the tools and capabilities needed for each worker. Produce a 15-item tool and boundary plan that answers:
1. What tool or capability is needed?
2. Which worker needs it?
3. Why is the tool needed?
4. Is access read-only or write-capable?
5. What actions are allowed?
6. What actions are prohibited?
7. What actions require human approval?
8. What data can the tool access?
9. What data must stay out of the tool?
10. What credentials or secrets are involved?
11. What least-privilege limit should apply?
12. What should be logged?
13. What should happen if the tool fails?
14. How should tool outputs be verified?
15. How should boundary violations be tested?

Successful Output: A tool permission map based on least privilege.

Step 8: Add Governance And Human Approval Gates

Place human review before consequential action.

Ask AI 
Identify every point in this workflow where governance or human approval is required. Produce a 15-item approval-gate plan that answers:
1. What actions have real-world consequence?
2. What external communications need approval?
3. What production changes need approval?
4. What spending or financial actions need approval?
5. What sensitive data requires special handling?
6. What legal, medical, HR, or security decisions need escalation?
7. What public claims need review?
8. What final deliverables need approval?
9. Who can approve each gate?
10. What evidence does the approver need?
11. What context must be preserved for approval?
12. What happens if approval is denied?
13. What happens if approval is delayed?
14. What audit trail is required?
15. How do we avoid meaningless rubber-stamp approval?

Successful Output: A governance map with clear stop points.

Step 9: Test, Observe, And Improve The System

Validate the system before trusting it with real work, then observe it after it runs.

Ask AI 
Create a testing, evaluation, and observability plan for this multi-agent workflow. Produce a 15-item validation plan that answers:
1. What normal test cases should pass?
2. What edge cases should be tested?
3. What unsafe requests should be tested?
4. What bad inputs should be tested?
5. What tool failures should be simulated?
6. What handoff failures should be simulated?
7. What stale memory cases should be tested?
8. What missed approval-gate cases should be tested?
9. What rubric defines quality?
10. What logs should be captured?
11. What traces should be captured?
12. What cost and latency signals matter?
13. What human feedback should be collected?
14. What regression tests should be created?
15. How will failures become improvement work?

Successful Output: A practical validation plan before trusting the system with real work.

Design Notes
  • Make the modal usable as a working guide, not just a reading experience.
  • Each collapsible step should be independently scannable.
  • Use copy buttons for each AI prompt if available.
  • Keep the prompt text in monospace/code styling.
  • Keep explanations short enough for a pop-up.
  • The page should emphasize that multi-agent systems are designed from work, contracts, handoffs, memory, tools, governance, and tests, not from agent names alone.

LangGraph Use for: Stateful orchestration, graph-based workflows, routing, supervisor patterns, memory-aware agent systems, durable task flow. Teaching angle: Strongest example for serious orchestration: nodes, edges, state, routing, retries, and human approval gates. LangGraph Supervisor Use for: Supervisor-agent patterns, delegation, centralized routing, controlled communication between specialized agents. Teaching angle: Shows how one coordinator can manage specialized agents without every agent talking to everything. CrewAI Use for: Roles, tasks, crews, agent collaboration, structured multi-agent work. Teaching angle: Maps well to “define the work before you build the agent” with role/task/crew vocabulary. CrewAI Examples Use for: Concrete examples and demos. Teaching angle: Practical workflows when students need examples instead of only abstract framework concepts. OpenAI Swarm Use for: Lightweight handoffs, agent transfer patterns, educational multi-agent examples. Teaching angle: Good for simple handoff mechanics. Frame as educational and lightweight, not the default production recommendation. Microsoft Agent Framework Use for: Enterprise agent architecture, multi-agent workflows, Python/.NET agent development, production-oriented patterns. Teaching angle: Shows where enterprise agent frameworks are heading. Microsoft AutoGen Use for: Historical and conceptual multi-agent patterns, agent conversations, collaborative workflows. Teaching angle: Useful conceptually, while noting Microsoft is moving toward Agent Framework as the newer direction. AWS Multi-Agent Orchestration With LangGraph Use for: Cloud deployment thinking, customer support examples, production architecture, AWS service integration. Teaching angle: Orchestration is infrastructure, services, routing, persistence, and operations, not only prompts. Agency Swarm Use for: Agency-style agent structures, organizational metaphors, specialized agent collaboration. Teaching angle: Useful for discussing whether agents should mirror departments, roles, workflows, or capabilities.
Class Framing
  • Swarm = lightweight handoffs.
  • CrewAI = roles, tasks, and teams.
  • LangGraph = stateful orchestration.
  • Microsoft Agent Framework = enterprise agent direction.
  • AWS guidance = production/cloud architecture.
  • Agency Swarm = organization-style agent design.
