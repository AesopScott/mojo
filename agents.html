Before Phase 0

What is an agent?

An agent is software that uses a model, context, tools, and rules to pursue a defined outcome. The important part is not that it chats. The important part is that it can make progress through a task loop with boundaries.

Define the concept See examples Continue to Phase 0

Agent shape

01 Goal

The outcome the agent is responsible for pursuing.

02 Loop

Observe, reason, act, check, and stop or escalate.

03 Boundary

What it may do alone, what needs approval, and what is forbidden.

A practical definition

For MAPS, define agents by their responsibility, operating loop, tools, and authority.

Responsibility

An agent owns a job, such as finding information, drafting a response, updating a ticket, or coordinating a handoff.

Context

An agent reads instructions, user input, files, memory, retrieved knowledge, tool results, or system state.

Action

An agent may call tools, ask questions, create artifacts, route work, request approval, or stop when the task is done.

Judgment

An agent chooses next steps from context and rules instead of only following a fixed script.

Boundary

An agent needs explicit permission limits, escalation points, failure behavior, and forbidden actions.

Proof

An agent should have observable success criteria, failure cases, evals, traces, or tests that prove it works.

Not every model call is an agent

The line matters because agents need design, permissions, evaluation, and observation.

Prompt

A one-time instruction that returns an answer or artifact. It does not own a task loop.

Workflow

A defined sequence of steps. It may use AI, but the path is mostly predetermined.

Agent

A bounded worker that can decide next actions, use tools, check progress, and escalate or stop.

Common single-agent examples

A single agent can still be useful when the job has one clear owner and a bounded outcome.

Research agent

Finds sources, compares claims, summarizes evidence, and reports uncertainty.

Search agent

Turns a question into searches, filters results, opens sources, and returns grounded answers.

Build agent

Reads a codebase, implements a scoped change, runs checks, and reports what changed.

Support agent

Triage requests, draft replies, look up account context, and escalate sensitive cases.

Calendar agent

Reads availability, proposes times, drafts invitations, and asks before sending or rescheduling.

Data agent

Answers business questions from approved datasets, explains queries, and flags data quality issues.

Common multi-agent examples

Multi-agent systems split a larger outcome across agents with different responsibilities.

System Agents Coordination pattern
Product feature delivery Research, planner, builder, reviewer, release agent. Planner decomposes work; builder implements; reviewer gates quality; release agent deploys with approval.
Customer support operations Intake, account lookup, policy checker, response drafter, escalation agent. Intake routes cases; specialist agents add context; escalation handles high-risk decisions.
Content production Researcher, outline agent, writer, editor, compliance checker, publisher. Artifacts pass through review gates before anything is published.
Sales enablement Lead researcher, account strategist, email drafter, CRM updater, approval agent. Research informs strategy; drafts require human approval before outreach or CRM changes.
Incident response Monitor, triage, diagnostics, comms, remediation planner, postmortem agent. Monitoring triggers triage; diagnostics gather evidence; humans approve remediation.

How this fits MAPS

MAPS separates the multi-agent project layer from the pipeline for building one agent.

M layer

Choose the system and roster

Define the overall outcome, decide which agents are needed, assign responsibilities, design coordination, and prove the full system.

System layer

APS layer

Build one agent well

Define, design, build, equip, evaluate, deploy, observe, and improve one agent or one agent capability.

Agent layer

Specification reference

The GSA-TTS AI Agent Specification Template is a useful reference for turning the word agent into a concrete, governable system component.

Identity and mandate

Define a stable handle, role, organizational placement, mandate, core responsibilities, and persona so humans and other agents know what the agent is for.

Architecture and reasoning

Name the agent architecture type, planning method, reasoning patterns, memory architecture, and learning mechanism before assigning complex work.

Actions and tools

Use an action index, tool manifest, and resource permissions to make the agent's capabilities explicit and enforce least privilege.

Communication

Specify communication protocols, data contracts, collaboration rules, and handoff formats so agents can coordinate predictably.

Governance

Document oversight, auditability, monitoring, validation, and escalation expectations so the agent can be trusted in operation.
